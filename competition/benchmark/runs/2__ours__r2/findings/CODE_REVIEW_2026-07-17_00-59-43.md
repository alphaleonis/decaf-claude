## Findings

### #1 🟠 High: Incomplete fix — intermediate-level member hiding still throws `AmbiguousMatchException`

| | |
|---|---|
| **File** | `src/Components/Forms/src/EditContextDataAnnotationsExtensions.cs:376` |
| **Category** | correctness / incomplete-fix |
| **Confidence** | 100 (corroborated ×4; validator: **confirmed**) |
| **Found by** | quick-reviewer (High), broad-reviewer (**Critical** — dissent), dotnet-reviewer (High), adversarial-reviewer (High) |

**Issue:** The fix resolves a model property in two steps:

```csharp
propertyInfo = cacheKey.ModelType.GetProperty(
    cacheKey.FieldName,
    BindingFlags.Public | BindingFlags.Instance | BindingFlags.DeclaredOnly);

if (propertyInfo is null)
{
    propertyInfo = cacheKey.ModelType.GetProperty(
        cacheKey.FieldName,
        BindingFlags.Public | BindingFlags.Instance | BindingFlags.FlattenHierarchy);
}
```

The `DeclaredOnly` first pass only disambiguates member hiding when the **model (leaf) type itself** redeclares the hidden `new` property. When a property is hidden at an **intermediate ancestor** and the leaf/model type does **not** redeclare it, `DeclaredOnly(leaf)` returns `null` and control falls through to the `FlattenHierarchy` fallback. `BindingFlags.FlattenHierarchy` only affects the visibility of **static** members up the hierarchy; with `Public | Instance` (no `Static`) it is inert — the fallback walks the same instance-member hierarchy as the pre-fix `GetProperty(name)` call and collects **two** same-named, differently-typed `PropertyInfo` candidates (e.g. `int Tag` on the mid class and `object Tag` on the base). Reflection does not dedupe them (their signatures differ — `new`-hiding is not an override, so they don't share a vtable slot), and with no return-type/`types` filter the default binder throws `AmbiguousMatchException` — the exact crash this PR exists to fix.

Because the throw happens **before** `_propertyInfoCache[cacheKey] = propertyInfo;`, nothing is cached and the exception recurs on **every** `NotifyFieldChanged` for that field. It is uncaught: `OnFieldChanged` (lines 79–100) calls `TryGetValidatableProperty` with no `try/catch`, so it propagates synchronously out of `NotifyFieldChanged` (Blazor Server: circuit teardown; WASM: unhandled error).

Reproducing hierarchy (leaf inherits an intermediate hide without redeclaring):

```csharp
class A     { public object Foo { get; set; } }
class B : A { public new int Foo { get; set; } }   // hides at intermediate level
class C : B { }                                     // model type; does NOT redeclare Foo
// FieldIdentifier(new C(), "Foo") → DeclaredOnly(C)=null → FlattenHierarchy(C) collects B.Foo + A.Foo → throws
```

The claim is confirmed by transitivity from the reproduced original bug: the candidate set reflection collects when querying leaf `C` is identical to querying intermediate `B` directly, and `typeof(B).GetProperty("Foo")` is the same shape as the confirmed `#27095` crash (`typeof(DerivedModelWithHiddenProperty).GetProperty("OrderID")` throws). The PR's own `ValidatesPropertyHiddenAtMultipleInheritanceLevels` test **does not** exercise this path — `DeepDerivedModel` (the instantiated leaf) itself declares `public new int Tag`, so `DeclaredOnly` succeeds on the first call and the vulnerable fallback is never reached (see Finding #2).

> Note: `dotnet` is not installed here, so the reviewer-nominated standalone reflection probe could not be executed. The finding is confirmed by (a) 4 independent finders, (b) an independent validator re-deriving the reference-source semantics, and (c) transitivity from the reproduced original bug.

**Fix:** Replace the flat `FlattenHierarchy` fallback with an explicit walk up the hierarchy, applying `DeclaredOnly` at each level so the binder is never handed more than one candidate at a time (mirrors "most-derived declaration wins" at any depth):

```csharp
propertyInfo = null;
for (var t = cacheKey.ModelType; t is not null; t = t.BaseType)
{
    propertyInfo = t.GetProperty(
        cacheKey.FieldName,
        BindingFlags.Public | BindingFlags.Instance | BindingFlags.DeclaredOnly);
    if (propertyInfo is not null)
    {
        break;
    }
}
```

The repo already solves the identical problem this way in `src/Components/Components/src/Reflection/MemberAssignment.cs` (`GetPropertiesIncludingInherited`, a per-level `DeclaredOnly` walk) — though that type lives in a different assembly and is not directly reusable from Forms.

### #2 🟡 Medium: No test exercises the intermediate-hiding case; a test name overstates coverage

**Issue:** `ValidatesPropertyHiddenAtMultipleInheritanceLevels` reads as proof that hiding across multiple inheritance levels is handled, but `DeepDerivedModel : MidLevelModelWithShadow : ModelWithHiddenBaseProperty` has the leaf `DeepDerivedModel` redeclare `public new int Tag`, so the assertion is satisfied entirely by the first `DeclaredOnly` call — the `FlattenHierarchy` fallback is never exercised in an ambiguous state. No test constructs a leaf that inherits a mid-level hide **without** redeclaring it, which is exactly the gap that lets Finding #1 ship unnoticed. (`MidLevelModelWithShadow` already sets up the differently-typed intermediate hide; only a non-redeclaring leaf is missing.)

**Fix:** Add the missing regression model + test:

```csharp
class LeafInheritingMidLevelShadow : MidLevelModelWithShadow { } // does NOT redeclare Tag

[Fact]
public void ValidatesPropertyHiddenByNonLeafAncestor()
{
    var model = new LeafInheritingMidLevelShadow();
    var editContext = new EditContext(model);
    editContext.EnableDataAnnotationsValidation(_serviceProvider);

    var field = new FieldIdentifier(model, "Tag");
    editContext.NotifyFieldChanged(field); // must not throw AmbiguousMatchException
}
```

Against the current code this test **fails** (throws), pinning Finding #1; after the Finding #1 fix it passes (resolves the most-derived `string Tag`, no attributes, empty messages).

### #3 🟡 Medium: No rationale comment for the two-step lookup or the dropped `BindingFlags.Static`

| | |
|---|---|
| **File** | `src/Components/Forms/src/EditContextDataAnnotationsExtensions.cs:368-379` |
| **Category** | knowledge-loss |
| **Confidence** | 100 (corroborated, incl. the knowledge specialist) |
| **Found by** | broad-reviewer (Medium), knowledge-reviewer (SHOULD ≈ High — dissent; self-downgraded from MUST because the regression is test-caught) |

**Issue:** The only comment ("DataAnnotations only validates public properties, so that's all we'll look for") predates this change and does not record three non-obvious facts that now live only in the commit/tests:
1. **Why** the lookup is split into `DeclaredOnly`-then-`FlattenHierarchy` — that `GetProperty(name, …FlattenHierarchy)` throws `AmbiguousMatchException` on differently-typed `new`-hidden members, so the declared-first probe is a workaround (see `#27095`).
2. **`BindingFlags.Static` was deliberately dropped.** The old `GetProperty(name)` default flags included `Static`; a static property sharing a field's name would previously have been matched and validated. Now such fields are always skipped (a genuine behavior change bundled into a fix whose title only mentions hidden members; pinned only by `IgnoresStaticProperty`). This is a **safe** change — instance-only resolution matches `propertyInfo.GetValue(model)` semantics and a static property routed to `Validator.TryValidateProperty` would previously have thrown — but the decision is unrecorded.
3. When the derived `new` shadow carries no validation attributes, the base's attributes (e.g. `[Required]`) are intentionally **not** applied (pinned only by `SkipsValidationWhenDerivedShadowHasNoAttributes`).

**Why it matters:** A future maintainer who doesn't know the reflection history may "simplify" this back to a single `GetProperty` call, silently reintroducing `#27095`. A code-local comment makes the constraint respectable on sight.

**Fix:** Add a rationale comment above the first `GetProperty` call, e.g.:

```csharp
// Resolve in two steps to avoid AmbiguousMatchException: when a derived model hides a base
// property with `new` (different signature), GetProperty(name, FlattenHierarchy) matches both
// candidates and throws (#27095). Look up the model type's own declared property first
// (most-derived wins, matching C# member access), then fall back to the inherited property.
// Static is intentionally excluded — DataAnnotations validates instance state only.
```
