## Findings

### #1 🔴 Critical: New inner read loop withholds output until the buffer fills or EOF — breaks `--line-buffered` / streaming stdin

| | |
|---|---|
| **File** | `crates/searcher/src/line_buffer.rs:419` |
| **Category** | correctness / data-loss (silent output-withholding) |
| **Confidence** | 100 (anchor) — validator-confirmed **and** confirmed against upstream ground truth |
| **Found by** | broad-reviewer (Critical), rust-reviewer (High), quick-reviewer (High) |

**Issue:** The PR replaces the single `read()` per `fill()` iteration with an inner loop that keeps reading until the *entire* free buffer is full or the reader returns `0` (true EOF), before any line-terminator scan or return to the caller:

```rust
let oldend = self.end;
while !self.free_buffer().is_empty() {
    let readlen = rdr.read(self.free_buffer())?;
    if readlen == 0 { break; }
    self.end += readlen;
}
let newbytes = &mut self.buf[oldend..self.end];
if newbytes.is_empty() { self.last_lineterm = self.end; return Ok(!self.buffer().is_empty()); }
// binary detection + rfind line terminator run only *after* the loop drains free_buffer
```

For a **streaming / blocking** source — a pipe from `tail -f` / `journalctl -f`, a TTY, or any producer that writes a line then pauses — once one short `read()` returns a complete line, the loop immediately issues another `read()` that *blocks* waiting for more data. Nothing is handed to `Core`/`Sink` (and therefore never flushed to output) until roughly a full buffer accumulates or the upstream closes its end. This silently defeats the `--line-buffered` flag, which exists precisely for streaming pipelines (`crates/core/flags/defs.rs` documents `tail -f something.log | rg foo --line-buffered | rg bar`). The pre-PR code returned as soon as a single `read()` produced a complete line.

**Ground-truth confirmation:** The ripgrep author reverted exactly this hunk 5 days after merge — upstream commit `d47663b1b454` ("searcher: fix regression with `--line-buffered` flag", **Fixes #3194**):

> *"in retrospect, this change is clearly contrary to how `--line-buffered` works. We specifically do _not_ want to wait around until the buffer is full. We want to read what we can, search it and move on."*
> *"So this reverts the first fix but leaves the second, which still keeps #3184 fixed and also fixes #3194 (the regression)."*

The revert restores the single `read()` per iteration in `line_buffer.rs` and the `glue.rs` byte counts, and **keeps** the independent `core.rs` roll change (Finding #4's subject). The regression shipped in ripgrep 15.0.0 (issue #3194: `journalctl -n5 -f | rg --line-buffered 'Oct'` produced no output until the pipe closed).

**Fix:** Revert the inner loop to a single `read()` per outer-loop iteration, returning to the caller as soon as any complete line is available. Keep the `core.rs` `roll()`/`before_context` change (it is correct and independent).

```rust
loop {
    self.ensure_capacity()?;
    let readlen = rdr.read(self.free_buffer())?;
    if readlen == 0 {
        self.last_lineterm = self.end;
        return Ok(!self.buffer().is_empty());
    }
    let oldend = self.end;
    self.end += readlen;
    let newbytes = &mut self.buf[oldend..self.end];
    // ... binary detection / lineterm search unchanged ...
}
```

### #2 🟡 Medium: Binary `Quit`-mode over-read — the inner loop drains the whole (grown) buffer before binary detection runs

| | |
|---|---|
| **File** | `crates/searcher/src/line_buffer.rs:419` |
| **Category** | performance (wasted I/O) |
| **Confidence** | 75 — validator-confirmed |
| **Found by** | performance-reviewer (Medium) |

**Issue:** Binary detection (`BinaryDetection::Quit`) runs only *after* the inner loop has fully drained the free buffer. When the backing buffer has grown large — via large `-A/--after-context` context, or a long unterminated line, forcing `ensure_capacity` doublings — a `Quit` binary byte appearing early in the newly-read region is not detected until up to the entire free buffer has been read. The pre-PR code checked for the binary byte after every single `read()`, bounding the worst-case over-read to one `read()` (~64K on stdin). The performance-reviewer measured **7,925,247 bytes read past a NUL byte** after forcing the buffer to grow to ~15.9 MB (vs. ≤64K pre-PR). This is directly relevant to the issue #3184 repro, which combines large `-A` with binary detection ("binary file matches").

Shares a root cause with Finding #1 (same line) and is resolved by the same upstream revert. Kept as a distinct finding because it is a different observable effect (wasted stream consumption vs. output latency).

**Fix:** Superseded by Finding #1's revert. Absent that, run the `Quit` byte check (and the line-terminator scan) after each individual `read()` inside the inner loop, breaking/truncating `self.end` as soon as the byte is seen, so worst-case over-read stays bounded by one `read()` regardless of buffer size.

### #3 🟡 Medium: No regression test exercises the short-read / streaming return behavior this PR changes

| | |
|---|---|
| **File** | `crates/searcher/src/line_buffer.rs` / `crates/searcher/src/searcher/glue.rs` (test modules) |
| **Category** | test-coverage |
| **Confidence** | 75 — corroborated ×2 (validation waived) |
| **Found by** | test-reviewer (Medium), broad-reviewer (Medium) |

**Issue:** Every "Read"-path test in the crate drives searches through `SearcherTester`, which hands `haystack.as_bytes()` (`&[u8]`) as the reader. `Read for &[u8]` copies `min(dest.len(), remaining)` in a single call and never blocks or returns a short read while data remains — so no existing test can model a reader that returns one line and then would block, which is exactly the condition that triggers the `--line-buffered` regression (Finding #1). This is *why* the regression shipped undetected by CI and was only caught by a live user (issue #3194). The `binary3`/`binary4` golden updates (`byte count` 262146 → 262142) reflect a binary-detection chunk-boundary shift for in-memory readers; they do **not** exercise the short-read/streaming path.

**Fix:** Add a throttling `Read` wrapper to `testutil.rs` (caps each inner `read()` to a small fixed size regardless of destination length) and a test asserting `fill()` returns as soon as one complete line is available (or that output is emitted per-line under `--line-buffered`) — directly guarding against regression of the streaming behavior.

### #4 🟢 Low: `roll()` comment omits *why* after-context lines need no preservation (the invariant that makes `before_context` correct)

| | |
|---|---|
| **File** | `crates/searcher/src/searcher/core.rs:190-206` |
| **Category** | knowledge-preservation |
| **Confidence** | 75 — validator-confirmed |
| **Found by** | knowledge-reviewer |

**Issue:** The change from `lines::preceding(buf, term, self.config.max_context())` to `...self.config.before_context)` is **correct** (all reviewers agree; it was kept upstream). But the added comment justifies only the *cost-skip* ("we only need to find the N preceding lines based on before context… skip this step when `before_context==0`"). It never states the load-bearing invariant that makes `before_context` (rather than `max_context()`) correct: after-context lines are emitted from data read *after* a match, and `after_context_left` is a `Core` counter that deliberately persists across `roll()` (which resets `last_line_counted`/`last_line_visited` but **not** `after_context_left`), so after-context resumes on the next fill and is never preserved by the backward-walking `preceding()`. Failure mode: a future maintainer "restores" `max_context()` for symmetry; correctness tests still pass (after-context output is unchanged), silently reintroducing the superlinear slowdown under large `-A`.

**Fix:** Extend the comment at the `preceding()` call, e.g.: *"We pass `before_context`, not `max_context()`: after-context lines are read after a match and re-emitted via `after_context_left`, which persists across `roll()` (it is not reset below), so after-context never needs preserving here — `max_context()` would over-preserve and cause superlinear slowdown under large `-A`."*

### Considered But Not Flagged (all agents)

- **`core.rs` `roll()`: `max_context()` → `before_context` correctness** — independently verified correct by quick, broad, spec-compliance, consistency, and rust reviewers, and by the fact that upstream **kept** this change when it reverted its sibling. `after_context_left` persists across `roll()`; when after-context is pending, `last_line_visited` dominates `max(context_start, last_line_visited)`, so the third arg to `preceding()` is moot in that case and only reduces cost when `after_context == 0`. No after-context lines are dropped at a buffer boundary. (Positive; the only residual is the comment nit in Finding #4.)
- **`binary3`/`binary4` byte-count shift 262146 → 262142** — intended consequence of binary detection running over a larger accumulated read chunk; the correctness-relevant `binary offset` values (262153 / 262149) are unchanged and cross-validated against the independent slice-reader path. Passing gate (77/77). Not a defect (and reverted upstream alongside the loop).
- **EINTR handling in the new inner loop** (consistency-reviewer, anchor 50; rust-reviewer refuted) — the loop propagates `io::ErrorKind::Interrupted` via `?` rather than retrying, unlike the crate's other read-until-full loop (`searcher/mod.rs:1005-1011`). Suppressed: below the confidence gate, **pre-existing** (the single-read `fill()` also used `?`, so not introduced by this diff — only the exposure window widens), and moot once the loop is reverted.
- **Extra harmless `read()` after EOF** (broad-reviewer, anchor 50 Low) — if the inner loop breaks on EOF after a partial read with no line terminator, the outer loop issues one more `read()` on the exhausted reader. Single extra syscall in a narrow edge case; negligible.
- **Inner read-loop lacks a "why fill the buffer" comment** (knowledge-reviewer, D2) — subsumed by Finding #1: the loop should be removed, so documenting it is moot.
- **Integer overflow / slice-bounds / aliasing in the new code** (rust-reviewer) — `self.end += readlen`, `&mut self.buf[oldend..self.end]`, repeated `free_buffer()` reborrows, and `.as_bytes_mut()` removal all verified safe (`oldend ≤ self.end` always; `free_buffer()` already returns `&mut [u8]`, so the dropped `.as_bytes_mut()` was redundant and makes the call more consistent with the `mod.rs` sibling).

## Reviewer's bottom line

The PR bundles **two independent changes**. The `core.rs` `roll()` optimization (use `before_context` instead of `max_context()`) is correct, well-reasoned, and is what actually fixes #3184 — it was kept upstream. The `line_buffer.rs` inner read loop *also* helped #3184 but introduced a Critical regression: it withholds output until the buffer fills, breaking `--line-buffered` streaming (Finding #1, plus the secondary Quit-mode over-read in #2). The author reverted exactly this half 5 days later (#3194). A regression test simulating short reads (Finding #3) would have caught it in CI.
