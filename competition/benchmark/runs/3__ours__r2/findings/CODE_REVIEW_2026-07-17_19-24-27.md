### #1 🔴 Critical: Canonical-wildcard matching in the general-purpose `CanCastTo` turns a documented `May` cast into `Must`, eliding JIT cast checks (type-safety hole in AOT shared generic code)

| | |
|---|---|
| **File** | `src/coreclr/tools/Common/TypeSystem/Common/CastingHelper.cs:426` (also `:474`, `:552`) + `src/coreclr/tools/Common/TypeSystem/Canon/CastingHelper.Canon.cs` |
| **Category** | soundness / type-safety |
| **Confidence** | 100 (2 independent finders + orchestrator code trace + validator CONFIRMED) |
| **Found by** | broad-reviewer (Critical), adversarial-reviewer (Critical); validated **confirmed** |

**Issue:** The PR inserts canonical-wildcard matching (`IsCanonEquivalent` / `IsCanonicalTypeArgMatch`) into the *shared, general-purpose* `CanCastTo(this TypeDesc, TypeDesc)` — specifically `CanCastToNonVariantInterface` (`:426`), `CanCastByVarianceToInterfaceOrDelegate` (`:474`), and `CanCastToClass` (`:552`) — not into a constraint-scoped helper. `CanCastTo` is consumed tree-wide, including by the JIT-EE interface `compareTypesForCast` in `CorInfoImpl.cs` (compiled into the real AOT/ReadyToRun/NativeAOT compiler via `ILCompiler.RyuJit → ILCompiler.TypeSystem`).

Traced end-to-end for `fromType = IFoo<__Canon>` (invariant interface), `toType = IFoo<string>`:

- `IsCanonicalTypeArgMatch(__Canon, string)` returns `true` because `string.IsGCPointer` is true (`CastingHelper.Canon.cs:37-38`), so `IsCanonEquivalent(IFoo<__Canon>, IFoo<string>)` returns `true`, so `CanCastToNonVariantInterface` returns `true`. **Before the PR this returned `false`** (the line was just `otherType.IsEquivalentTo(thisType)` — verified via `git show`).
- In `compareTypesForCast` (`CorInfoImpl.cs:2915-2995`), this pair lands in the `fromType`-shared / `toType`-unshared interface branch (`:2939`). With `canCast == true` (now), execution short-circuits at `:2950-2952` to `result = TypeCompareState.Must` — **never reaching** the careful `else-if` chain (`:2954-2979`) whose own comment documents this exact case as `May`:
  ```
  //    IFoo<__Canon> -> IFoo<string>     May
  ```

`TypeCompareState.Must` is the JIT-EE contract signal that the cast will always succeed, so the JIT elides the runtime type check. Because `IFoo<__Canon>` is the shared-code stand-in for *every* reference-type instantiation of `IFoo<T>`, in `M<T>(IFoo<T> x)` compiled as shared canonical code an `x is IFoo<string>` / `(IFoo<string>)x` is now treated as always-true even when the real `T` is (e.g.) `object` — the cast that should fail succeeds, handing out a mistyped reference. That is type confusion in AOT-compiled code, the inverse hazard of the (false-negative) bug this PR set out to fix.

**Fix:** Scope the canonical-wildcard reasoning to the callers that need permissive "might match, re-validated at runtime" semantics (constraint pre-validation), and keep it out of the shared `CanCastTo` paths — e.g. expose a separate `CanCastToConsideringCanonAsWildcard` helper called explicitly from the constraint/dataflow sites, rather than editing `CanCastToInternal` / `CanCastToNonVariantInterface` / `CanCastToClass` / `CanCastByVarianceToInterfaceOrDelegate` in place. If the shared predicate must change, audit every `CanCastTo` consumer for soundness under the new semantics and **update the now-stale invariant comment in `CorInfoImpl.cs:2954-2967`**, which still documents the pre-PR `IFoo<__Canon> -> IFoo<string> May` behavior (a doc-vs-code contradiction folded into this finding).

### #2 🟠 High: `MakeGenericMethodSite` did not receive the `NormalizeInstantiation()` fix its twin `MakeGenericTypeSite` got, leaving the same "missing native code or metadata" failure reachable via `MethodInfo.MakeGenericMethod`

| | |
|---|---|
| **File** | `src/coreclr/tools/aot/ILCompiler.Compiler/Compiler/Dataflow/HandleCallAction.cs:760-774` (vs. the fixed `:783-794`) |
| **Category** | completeness / data-loss (missing metadata → runtime `NotSupportedException`) |
| **Confidence** | 90 (2 finders + validator CONFIRMED against a specific refutation) |
| **Found by** | broad-reviewer (Critical), quick-reviewer (High); validated **confirmed** |

**Issue:** `MakeGenericTypeSite.InstantiateDependencies` was changed to normalize the instantiation shape before checking constraints:
```csharp
TypeDesc instantiatedType = _type.InstantiateSignature(...);
instantiatedType = instantiatedType.NormalizeInstantiation();   // new
if (instantiatedType.CheckConstraints(...)) RootingHelpers.TryGetDependenciesForReflectedType(...);
```
The structurally identical `MakeGenericMethodSite.InstantiateDependencies` (`:760-774`) has **no** such normalization. `MethodDesc.InstantiateSignature` (`MethodDesc.cs:759-760`) computes the owning type via the same `TypeDesc.InstantiateSignature` primitive, so `instantiatedMethod.OwningType` can end up in the same denormalized shape (e.g. `Foo<object, __Canon>`).

The refutation offered by three reviewers — that `RootingHelpers.TryGetDependenciesForReflectedMethod` canonicalizes internally via `GetCanonMethodTarget(Specific)` (`RootingHelpers.cs:189`) — was checked and does **not** hold: that internal canonicalization runs only *after* the `CheckConstraints` gate at `HandleCallAction.cs:770`, which executes on the un-normalized `OwningType`. `MethodDesc.CheckConstraints` calls `method.OwningType.CheckConstraints` (`TypeSystemConstraintsHelpers.cs:208-211`), and `CanCastToConstraintWithCanon` (`TypeSystemConstraintsHelpers.Canon.cs:35`) grants the wildcard pass **only** when the instantiation param *is* a canonical definition type — a concrete stand-in like `object` left in place because normalization never ran gets no exemption and is checked structurally, so a constraint that `__Canon` would trivially satisfy can be spuriously rejected. A rejected `CheckConstraints` skips rooting entirely → the exact "missing native code or metadata" symptom of #126604, reachable through `MethodInfo.MakeGenericMethod`. No `MakeGenericMethod` regression test was added.

**Fix:** Apply equivalent normalization on the method side before `CheckConstraints` — normalize `instantiatedMethod.OwningType` and re-resolve the method onto it (the codebase has `FindMethodOnExactTypeWithMatchingTypicalMethod` for this), and add a `MakeGenericMethod` counterpart to `TestMakeGenericConstrainedDataflow`. (There is no `MethodDesc.NormalizeInstantiation()` overload today, so this needs the owning-type route.)

### #3 🟡 Medium: The intentional one-directional asymmetry of `IsCanonicalCastTarget` is undocumented, inviting an unsound "symmetrizing" edit

| | |
|---|---|
| **File** | `src/coreclr/tools/Common/TypeSystem/Canon/CastingHelper.Canon.cs:33` |
| **Category** | knowledge-preservation / comprehension-risk |
| **Confidence** | 75 |
| **Found by** | knowledge-reviewer (High → normalized to Medium) |

**Issue:** `IsCanonicalCastTarget` inspects only `otherType` (the cast *target*) for canon-ness and deliberately never treats a canon *source* (`thisType`) as a wildcard. The sibling method 12 lines below, `IsCanonicalTypeArgMatch`, checks *both* directions, and source-canon-as-wildcard lives only in `TypeSystemConstraintsHelpers.Canon.cs:CanCastToConstraintWithCanon` (sound there only because the runtime re-validates the concrete type). None of this rationale is stated in `CastingHelper.Canon.cs`. A maintainer fixing an unrelated canon-cast issue could "symmetrize" `IsCanonicalCastTarget` to match the adjacent two-directional method and make general casting unsound — and the reasoning that would stop that edit lives across a file boundary they need not read. (Related to #1: the arg-position wildcard in the *same* file is already the source of a confirmed soundness hole, so this area is genuinely fragile.)

**Fix:** Add a comment to `IsCanonicalCastTarget` stating the asymmetry is intentional — only the cast *target* being canonical is a wildcard; a canonical *source* is deliberately not, because that would make general casts unsound; source-canon-as-wildcard is valid only during constraint pre-validation (see `CanCastToConstraintWithCanon`).

### #4 🟡 Medium: Linking the real `CastingHelper.Canon.cs` into `System.Private.TypeLoader` extends the wildcard casting behavior into the runtime type loader's GVM resolution

| | |
|---|---|
| **File** | `src/coreclr/nativeaot/System.Private.TypeLoader/src/System.Private.TypeLoader.csproj:120` |
| **Category** | design / boundary-violation |
| **Confidence** | 50 · marked **unvalidated** (runtime GVM path not independently traced) |
| **Found by** | design-reviewer (Medium), broad-reviewer (Low) |

**Issue:** The runtime type loader compiles the real `CastingHelper.Canon.cs` (not the `NonCanon` stub), so its only `CanCastTo` consumer — variant GVM dispatch resolution in `TypeLoaderEnvironment.GVMResolution.cs:236` (`currentIfaceType.CanCastTo(declaringType)`), where `currentIfaceType` can be canonical in shared generics — now participates in the new wildcard semantics. That can change which interfaces are treated as variant-compatible and therefore which GVM target is selected (or trigger a Diamond/ambiguous/Reabstraction throw). This is a **runtime** behavior change beyond the PR's stated compile-time ILC-dataflow intent, uncommented and unexercised by the added tests. Given #1 confirms the wildcard `CanCastTo` is unsound for cast-elision in the compiler, the runtime GVM consumer deserves an explicit soundness check.

**Context — this was a deliberate, reviewed decision.** In the PR thread, jkotas asked "Do we actually need the real `CastingHelper.Canon.cs` in the runtime type loader?"; the author replied it was the simpler rule ("if you include a `.Canon.cs`, you include the `.Canon.cs` files") and offered to use the `NonCanon` stub instead, and jkotas accepted the Canon inclusion. So this is a known trade-off, not an oversight — but the GVM-dispatch consequence was not traced in that exchange.

**Fix:** Decide and document the intent. If the runtime type loader must not change GVM variant-dispatch behavior, link `CastingHelper.NonCanon.cs` there instead (the conservative choice preserves prior runtime behavior). If the wildcard behavior is wanted at runtime, add a comment at the companion include and a GVM-resolution test over canonical instantiations that pins the intended dispatch outcome.

### Considered But Not Flagged (all agents)

**Refuted by validators:**
- **Devirtualization unsoundness** (adversarial-reviewer, was High): the broadened `CanCastTo` does reach `DevirtualizationManager` (via `ILCompiler.Compiler → ILCompiler.TypeSystem`), but no wrong-target scenario survives — the pre-existing `MULTIPLE_IMPL` ambiguity block independently guards target selection, `resolveVirtualMethod` rejects a bare-canonical `objType` before resolution, a further backstop rejects a canonical resolved impl, and the `CanCastTo` call is only a bail-out gate (the target is computed by separate precise routines on the concrete `implType`). **Refuted.**
- **Unbounded recursion / missing `StackOverflowProtect` in `IsCanonicalTypeArgMatch`/`IsCanonEquivalent`** (dotnet-reviewer, was Medium): every recursive edge descends into a strictly-smaller structural sub-component (generic argument or `ParameterizedType` element); `InstantiatedType` is built bottom-up and is acyclic by construction, so no cycle is structurally possible. The sibling `CanCastByVarianceToInterfaceOrDelegate` needs `StackOverflowProtect` because it traverses `RuntimeInterfaces`/base chains (sideways, revisitable) — a different recursion shape. The deep-but-finite-nesting concern is pre-existing and not what `CastingPair`/`StackOverflowProtect` addresses (that's `IsGenericDepthGreaterThan`). **Refuted.**

**Ruled out on the merits (representative):**
- `IsCanonEquivalent` indexing `thisInst[i]`/`otherInst[i]` guarded only by `thisInst.Length` — safe: `HasSameTypeDefinition` fixes generic arity, so lengths always match; the `Length == 0` guard short-circuits non-generic types (multiple reviewers).
- Wildcard making a genuinely-invalid *constraint* pass (inverse of the fixed bug) — the constraint gate only over-approximates the dependency graph (roots an unneeded template); the runtime re-validates the concrete instantiation, so no runtime fault results from the constraint path itself. (The unsoundness is in the codegen `CanCastTo` consumer — finding #1 — not here.)
- Partial-class **build completeness** — verified by whole-repo grep: exactly 3 projects compile `CastingHelper.cs` (`ILCompiler.TypeSystem` → Canon, `System.Private.TypeLoader` → Canon, `ILVerification` → NonCanon) and 2 compile `TypeSystemConstraintsHelpers.cs` (`ILCompiler.TypeSystem` → Canon, `ILVerification` → NonCanon); `System.Private.TypeLoader` doesn't compile the constraints helper, so it needs no companion for it. No missing/duplicate companion; no CS0103/CS0111 risk (broad, dotnet, consistency, spec, prior-feedback all independently confirmed).
- `_ => throw new UnreachableException()` in `IsSpecialTypeMeetingConstraint` — genuinely unreachable; all 3 call sites pass one of the 3 handled constraint flags (dotnet).
- Prior PR feedback — the jkotas consistency rule is upheld in the final code; all four copilot-bot suggestions were either addressed (unused locals/usings not present in final code) or correctly declined (semicolon-terminated empty type declarations are valid C# already compiling in the same project via `DeadCodeElimination.cs`; `using System.Diagnostics;` in the `.Canon.cs` file is needed for `UnreachableException`) (prior-feedback-reviewer).
