## Considered But Not Flagged

**[ADV_ABUSE] Self-referential / deeply-nested type → infinite or pathological recursion in `IsCanonicalTypeArgMatch`/`IsCanonEquivalent`** — REFUTED (termination proven).
Both functions recurse *strictly structurally*: `IsCanonEquivalent` iterates `thisType.Instantiation` and calls `IsCanonicalTypeArgMatch(thisInst[i], otherInst[i])` on instantiation *arguments* (proper subcomponents); the `ParameterizedType` branch recurses on `paramType.ParameterType` (the element — a proper subcomponent). A constructed `TypeDesc` is a finite tree; you cannot materialize a type whose `Instantiation` contains itself (the inner must exist before the outer). The issue's `IRequest<TRequest,TResponse> where TRequest : IRequest<...>` is a *constraint* self-reference — the instantiated instance `IRequest<Foo,Bar>` still has a finite `Instantiation` `[Foo,Bar]`. Depth ≤ input type nesting depth, identical to the pre-existing `IsEquivalentTo`/`CanCastTo` structural recursion on the same types. No cycle, no unbounded stack. (`CastingHelper.Canon.cs:24-92`)

**[ADV_ABUSE] Hostile DAG-shaped type → superlinear traversal (no memoization)** — not flagged (not emergent / no worse than baseline). A shared interned subtree S appearing N times down a spine costs O(N·|S|) because the new predicates carry no memo table. But the *existing* `IsEquivalentTo`/`CanCastTo` comparison code has the identical characteristic on the identical types (`StackOverflowProtect` guards cycles, not cost). This change adds no worse asymptotic behavior than the type-comparison code already reachable on the same inputs. Anchor 25.

**[ADV_ASSUMPTION] `IsCanonEquivalent` indexes `otherInst[i]` after only checking `thisInst.Length==0` → IndexOutOfRange on arity mismatch** — REFUTED. The guard is `HasSameTypeDefinition` (line 46: `GetTypeDefinition() == other.GetTypeDefinition()`). Two types sharing a type definition necessarily have the same generic arity, hence `thisInst.Length == otherInst.Length`. I checked the awkward cases (open-definition vs constructed, nested generics, MD-arrays of differing rank): differing-arity types never share a definition, and arrays short-circuit via `thisInst.Length==0`. Even the open-def-vs-instantiated case has equal lengths and safely returns false. No index path is reachable with mismatched lengths. (`CastingHelper.Canon.cs:69-88`, `TypeDesc.cs:631`)

**[ADV_ASSUMPTION] Pointer/byref/function-pointer instantiation args violate the XML-doc "not valid" assumption → wrong match or crash** — REFUTED. `FunctionPointerType` is not a `ParameterizedType`, so it falls through every branch to `return false`. Pointer/byref leaves are correctly rejected by the `IsGCPointer` checks (`int*`.IsGCPointer is false → no `__Canon` match). The pointer-element-array case (`int*[]`, which slips past `CheckValidInstantiationArguments`) recurses into `int*` vs `__Canon` and correctly returns false. No branch produces a wrong TRUE or an unhandled cast.

**[ADV_COMPOSITION] Swapped argument order at call sites → silent under/over-match** — REFUTED. `IsCanonEquivalent` and `IsCanonicalTypeArgMatch` are symmetric (they test canon on both sides), so their two symmetric call sites (`CanCastToNonVariantInterface`, `CanCastToClass`, variance loop) are order-insensitive. The two *asymmetric* predicates are called with the correct direction: `IsCanonicalCastTarget(thisType, otherType)` in `CanCastToInternal` uses `thisType.IsGCPointer` for target=`__Canon` (source must be a ref type — correct); `CanCastToConstraintWithCanon(instantiationParam, instantiatedConstraintType)` treats the param as wildcard / checks the constraint side — correct. I verified `__UniversalCanon` is a `ValueType` (`IsGCPointer==false`) and `__Canon` is a `Class` (`IsGCPointer==true`), so `IsCanonicalCastTarget(__UniversalCanon, __Canon)` correctly returns false (a value-representing canon cannot cast to the ref-only canon).

**[ADV_COMPOSITION/CASCADE] Relaxed canon match skips a needed constraint check → invalid code precompiled** — analyzed, harm direction is benign. Every change is strictly *more* permissive (`|| <new>` / `if(new) continue;`), so `CheckConstraints` can only flip false→true, never true→false. Over-approval roots a canonical *shared* body that is unused when the concrete instantiation is invalid (reflection `MakeGenericType` still throws `ArgumentException` at runtime as it should). The harmful direction — under-rooting a needed artifact → runtime `MissingRuntimeArtifact` — is not made worse by this change. Over-matching for `__Canon`/`__UniversalCanon` is by-design (canonical instantiations are placeholders validated at concrete construction), and concrete-vs-concrete pairs with no canon still return false, so the relaxation never leaks into non-canonical matching.

**[ADV_COMPOSITION] `NormalizeInstantiation()` in `HandleCallAction` flips a constraint result (`Foo<object,__Canon>` → `Foo<__Canon,__Canon>` masks a failing `object`-vs-`IFoo` check)** — analyzed, benign + intended. `NormalizeInstantiation` is a no-op unless the instantiation *already* contains a canon type (`IsCanonicalSubtype(Any)`), i.e. it is already runtime-determined. It only converts reference-type args to `__Canon`, preserving special-constraint outcomes (object and __Canon both satisfy `class`, neither satisfies `struct`). It can turn a type-constraint check from false→true for a literal `object` arg sitting beside a `__Canon`, but that only over-roots a shared body (benign). Critically, this call restores the invariant the scanner already asserts downstream — `Debug.Assert(type.NormalizeInstantiation() == type)` at `ILScanner.cs:770/777/785` — so *omitting* it (the prior behavior) was the latent defect (assert/denormalized rooting). The added call is the fix, in the safe direction.
