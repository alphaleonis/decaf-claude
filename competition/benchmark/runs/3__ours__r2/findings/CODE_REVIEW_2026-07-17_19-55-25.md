## Findings

### #1 🔴 Critical: Canonical-wildcard matching added to the general-purpose `CanCastTo` flips the JIT-EE `compareTypesForCast` result for `IFoo<__Canon> → IFoo<string>` from `May` to `Must`, eliding a needed runtime cast check

| | |
|---|---|
| **File** | `src/coreclr/tools/Common/TypeSystem/Common/CastingHelper.cs:426` (also `:474`, `:552`) + `src/coreclr/tools/Common/TypeSystem/Canon/CastingHelper.Canon.cs`; consumer `src/coreclr/tools/Common/JitInterface/CorInfoImpl.cs:2946-2967` (`compareTypesForCast`) |
| **Category** | soundness / type-safety |
| **Confidence** | 100 for the doc-vs-code contradiction and the reachable `May→Must` flip (git-verified + opus validator CONFIRMED after adversarial refutation); the downstream JIT elision itself lives in the C++ backend not in this repo, so the ultimate type-confusion consequence is a strong static inference, not runtime-reproduced |
| **Found by** | orchestrator (Step 7 cross-review + trace); adversarial-reviewer surfaced the surface (Probe Request #1, unconfirmed); opus finding-validator confirmed |

**Issue:** The PR inserts canonical-wildcard matching (`IsCanonEquivalent` / `IsCanonicalTypeArgMatch`) into the *shared, general-purpose* `TypeDesc.CanCastTo` — specifically `CanCastToNonVariantInterface` (`CastingHelper.cs:426`), `CanCastByVarianceToInterfaceOrDelegate` (`:474`), and `CanCastToClass` (`:552`) — not into a constraint-scoped helper. `CanCastTo` is consumed compiler-wide, including by the JIT-EE callback `compareTypesForCast` in `CorInfoImpl.cs`, which is compiled into the **real** NativeAOT (`ILCompiler.RyuJit`) and crossgen2 (`ILCompiler.ReadyToRun`) compilers — both `ProjectReference` `ILCompiler.TypeSystem`, which this PR wired the relaxed `CastingHelper.Canon.cs` into (only ILVerification gets the `NonCanon` stub).

Verified end-to-end for `fromType = IFoo<__Canon>` (shared, invariant interface), `toType = IFoo<string>` (unshared):
- `IsCanonicalTypeArgMatch(__Canon, string)` returns `true` (`__Canon` is a Specific canonical definition type → returns `string.IsGCPointer`), so `IsCanonEquivalent(IFoo<__Canon>, IFoo<string>)` → `true`, so `IFoo<__Canon>.CanCastTo(IFoo<string>)` now returns **true**. Pre-PR it returned **false** (git-verified: the line was `otherType.IsEquivalentTo(thisType)` only).
- In `compareTypesForCast`, the `fromType.IsCanonicalSubtype(Any) && !toType.IsCanonicalSubtype(Any)` + `toType.IsInterface` branch runs `bool canCast = fromType.CanCastTo(toType); if (canCast) result = Must;`. Post-PR `canCast == true` → **`result = Must`**. Pre-PR `canCast == false` fell through to `toType.HasInstantiation` → `result = May`.
- The in-code comment block at `CorInfoImpl.cs:2954-2967` still documents this exact case as `May` and explains the else-if chain exists precisely to catch `CanCastTo`'s pre-PR *false negatives*:
  ```
  //    __Canon       -> IBar             May
  //    IFoo<__Canon> -> IFoo<string>     May
  //    IFoo<__Canon> -> IBar             MustNot
  ```
  The `#if READYTORUN` softening only downgrades `MustNot`→`May`, never `Must`. The PR does not touch `CorInfoImpl.cs`.

`TypeCompareState.Must` is the JIT-EE contract signal that a cast will always succeed, so the JIT may elide the runtime cast check. Because `IFoo<__Canon>` is the shared-code stand-in for *every* reference-type instantiation of `IFoo<T>`, in `M<T>(IFoo<T> x)` compiled as shared canonical code, `(IFoo<string>)x` becomes always-true even when the real `T` is (e.g.) `object` — the cast that should throw `InvalidCastException` succeeds, yielding a mistyped reference. That is type confusion in AOT-compiled code: the inverse hazard of the false-negative bug this PR set out to fix. At minimum, the change introduces a live doc-vs-code contradiction the PR failed to update.

**Fix:** Keep the canonical-wildcard reasoning out of the shared `CanCastTo` paths — expose a separate `CanCastToConsideringCanonAsWildcard` (or a `bool treatCanonAsWildcard` parameter) called explicitly from the constraint/dataflow pre-validation sites, rather than editing `CanCastToNonVariantInterface` / `CanCastByVarianceToInterfaceOrDelegate` / `CanCastToClass` in place. If the shared predicate must change, audit every `CanCastTo` consumer for soundness under the new semantics and update the now-stale `CorInfoImpl.cs:2954-2967` comment and its `else if (canCast) result = Must` logic (e.g. gate the `Must` short-circuit on `!fromType.IsCanonicalSubtype(Any)`).

### #2 🟠 High: Sibling `MakeGenericMethodSite` lacks the `NormalizeInstantiation()` that fixed `MakeGenericTypeSite` — the same bug class (#126604) can recur via `MakeGenericMethod`

| | |
|---|---|
| **File** | `src/coreclr/tools/aot/ILCompiler.Compiler/Compiler/Dataflow/HandleCallAction.cs:770` (vs the fix at `:786-788`) |
| **Category** | correctness / fix-completeness |
| **Confidence** | 50 (quick-reviewer) → **validation confirmed** the structural asymmetry and refuted the proposed mitigation; not runtime-reproduced |
| **Found by** | quick-reviewer (High); broad-reviewer investigated and initially dismissed it — dismissal refuted by validation |

**Issue:** The fix for `MakeGenericTypeSite` inserts `instantiatedType = instantiatedType.NormalizeInstantiation();` after `InstantiateSignature` and *before* `CheckConstraints` ("InstantiateSignature could end up with a denormalized shape (`Foo<object, __Canon>`) so normalize"). The sibling `MakeGenericMethodSite` builds `instantiatedMethod` via the identical `InstantiateSignature` pattern (whose owning type comes from the same `TypeDesc.InstantiateSignature`, so it can take the identical denormalized shape) and calls `CheckConstraints` at `:770` **with no normalization**. `MethodDesc.CheckConstraints` feeds that unnormalized owning-type + method instantiation straight into `VerifyGenericParamConstraint` / `CanCastToConstraintWithCanon` — the exact logic this PR patched for the type case.

broad-reviewer argued this is harmless because the method path later calls `GetCanonMethodTarget(CanonicalFormKind.Specific)`. **Validation refuted that**: `GetCanonMethodTarget` is invoked only inside `RootingHelpers.TryGetDependenciesForReflectedMethod` (`RootingHelpers.cs:189`), which `HandleCallAction.cs` calls at `:771` — *strictly after* `CheckConstraints` already returned at `:770`. The later normalization cannot protect the earlier constraint check. There is also no regression test for the `MakeGenericMethod` path (the new smoke test exercises `MakeGenericType` only).

Residual uncertainty (why 50, not higher): the asymmetry and the refuted-mitigation are code-confirmed, but a concrete program that drives a denormalized method instantiation into this check and produces a user-visible `NotSupportedException` was not reproduced (the compiler cannot be built here). This is a completeness gap in the fix, not a defect in what was implemented.

**Fix:** Apply the same normalization on the method path before the constraint check, and add a `MakeGenericMethod` regression case. (Confirm the exact method-level normalize API; the point is that `CheckConstraints` must see a normalized shape, as the type path already ensures.)

### #3 🟡 Medium: `.Canon.cs`/`.NonCanon.cs` per-project selection is an undocumented, unguarded manual convention

| | |
|---|---|
| **File** | `src/coreclr/tools/Common/TypeSystem/Common/CastingHelper.NonCanon.cs`, `.../TypeSystemConstraintsHelpers.NonCanon.cs`, and the 3 project files that pick a variant (`ILCompiler.TypeSystem.csproj:134`, `System.Private.TypeLoader.csproj:120`, `ILVerification.projitems:378`) |
| **Category** | knowledge-preservation / design (evolution readiness) |
| **Confidence** | 100 (multi-finder; validation confirmed) |
| **Found by** | broad-reviewer (Medium), knowledge-reviewer (High/SHOULD — dissent), design-reviewer (Medium, conf 50) |

**Issue:** The canon-aware methods (`IsCanonicalCastTarget`, `IsCanonicalTypeArgMatch`, `IsCanonEquivalent`, `IsSpecialTypeMeetingConstraint`, `CanCastToConstraintWithCanon`) are ordinary `private static` methods (not C# `partial` methods) defined in exactly one of two mutually-exclusive files — `.Canon.cs` (real logic) or `.NonCanon.cs` (stubs returning `false`). Which one a project compiles is decided solely by its `.csproj`/`.projitems` `<Compile Include>` list. The selection criterion ("does this project run reflection/dataflow analysis over `__Canon`/`__UniversalCanon`?") is written nowhere — no comment at the call sites, in the stub file headers, or beside the `<Compile>` entries. Validation confirmed the failure mode is **asymmetric**: omitting both variants, or wiring `.Canon.cs` into a project lacking canonical-type support, fails to compile (loud, safe — `IsCanonicalDefinitionType` lives only in `TypeSystemContext.Canon.cs`); but wiring the `.NonCanon.cs` stub into a project that *does* process canonical types **compiles cleanly and silently under-approximates** (returns `false` where it should match), reintroducing exactly the bug class this PR fixes. Current wiring is verified correct across all three consumers — this is a latent-maintainability finding, not a present bug.

Severity dissent: knowledge-reviewer rated this High (comprehension debt); broad and design rated Medium (current code correct; the wrong-choice case requires a future maintainer actively mis-wiring). Consolidated at **Medium** — the current wiring is correct and multi-reviewer-verified.

**Fix:** Add a one-line comment atop each `.NonCanon.cs` (and/or beside the variant `<Compile Include>` entries) stating the "compile exactly one variant" rule and when to pick which. Optionally add `Debug.Assert(!input.IsCanonicalSubtype(Any))` in the stubs so a wrong-variant miswiring fails loudly in debug builds (design-reviewer's suggestion).

### #4 🟡 Medium: Relaxed canon-aware `CanCastTo` is compiled into the NativeAOT **runtime** type loader and can reach a real GVM variant-dispatch decision

| | |
|---|---|
| **File** | `src/coreclr/nativeaot/System.Private.TypeLoader/src/System.Private.TypeLoader.csproj:120` → runtime call site `.../TypeLoaderEnvironment.GVMResolution.cs:236` |
| **Category** | design / cross-cutting correctness |
| **Confidence** | 50 (design-reviewer) — validation confirmed the branches are *live* in the type loader and the site is plausibly reachable; a wrong-dispatch was not demonstrated |
| **Found by** | design-reviewer (Medium) |

**Issue:** The PR's relaxations make `X.CanCastTo(__UniversalCanon)` return `true`, `X.CanCastTo(__Canon)` return `X.IsGCPointer`, and same-definition interface/base types with canon-vs-concrete type args match (all previously `false`). The type loader `.csproj` compiles the **real** `CastingHelper.Canon.cs` (not the stub), and validation confirmed `TypeLoaderTypeSystemContext.SupportsCanon => true` with `__Canon` bound to a genuine runtime handle — so the new branches are live at runtime. At `GVMResolution.cs:236`, variant generic-virtual-method dispatch does `variantDispatch && currentIfaceType.CanCastTo(declaringType)` — a genuine runtime dispatch decision with no compile-time-then-runtime re-validation beneath it. The PR's safety rationale ("over-permissive only over-roots at compile time; runtime re-validates with concrete types") does not cover this runtime consumer, so the relaxation could, in principle, alter which GVM slot is selected. (This is the runtime sibling of Critical #1: the same over-permissive `CanCastTo` reaching a *runtime dispatch* rather than a *codegen* decision.)

**Mitigating context (why Medium / confirm-monitor):** reviewer **jkotas asked this exact question on the PR** — *"Do we actually need the real CastingHelper.Canon.cs in the runtime type loader?"* — and the author replied *"Probably not right now... I can make it include .NonCanon.cs here and we can fix the bug if this ever becomes a bug,"* which jkotas approved. So the type-loader inclusion is a conscious, expert-accepted decision. The thread debated whether it's *needed*, not specifically whether it *changes GVM dispatch* — that narrower question remains open.

**Fix:** Confirm (ideally in a comment at `GVMResolution.cs:236` or a test) that canon-aware variant matching yields the correct slot under shared generics there. If the runtime dispatch needs stricter pre-PR semantics, route it through a distinct non-relaxed casting entry point so the compile-time and runtime contracts can't drift.
