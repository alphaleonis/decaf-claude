## Adversarial Review

### Summary

The `roll()` correctness change (before_context vs max_context) holds up under scrutiny — `last_line_visited` correctly protects un-emitted after-context and the context-separator logic, so no output regression there. However, the `line_buffer.rs` inner read loop introduces a real, documented-use-case-breaking latency regression on streaming stdin, and the entire PR ships with **zero new tests** for behavior that is subtle and easy to regress. The biggest concern: `tail -f log | rg foo --line-buffered` — an example ripgrep's own `--line-buffered` docs advertise — will no longer emit matches promptly.

### Findings

#### High

- **[other]** New inner read loop batches until the buffer is full, defeating line-buffered streaming input — `crates/searcher/src/line_buffer.rs:419-425`
  - **What's wrong/missing:** The old `fill()` did one `read()` per pass and returned `Ok(true)` as soon as the read completed at least one line (`rfind_byte(lineterm)` → return). The new inner loop `while !self.free_buffer().is_empty() { readlen = rdr.read(...)?; if readlen == 0 { break }; ... }` has **no early exit when a complete line is available** — it only breaks on `Ok(0)` (EOF). For a pipe that trickles data and stays open, the first `read()` returns a burst (e.g. one log line), the buffer is nowhere near full, so the loop calls `read()` again and **blocks** waiting to fill up to `DEFAULT_BUFFER_CAPACITY` (64 KB). Already-read complete lines sit unsearched until 64 KB accumulates or the pipe closes.
  - **Why it matters:** This directly contradicts documented behavior. `crates/core/flags/defs.rs:3562-3566` advertises `--line-buffered` for exactly `tail -f something.log | rg foo --line-buffered | rg bar`. But `--line-buffered` (verified at `crates/core/flags/hiargs.rs:839`) only controls **output** flushing — it does not change this input path. So a match can't be *flushed* until it's *found*, and it won't be found until `fill()` returns, which now blocks until the input buffer fills. On a low-volume log, matches are delayed indefinitely (until ~64 KB of new data or the stream ends — which for `tail -f` is never). This is a functional regression for interactive/streaming pipelines, not covered by any test and not called out in the CHANGELOG (which only advertises a perf win).
  - **Fix:** Restore per-line responsiveness: break out of the inner loop once `newbytes` (or the accumulated unsearched region) contains a line terminator, e.g. add `if self.free_buffer_has_lineterm() { break }` or check after each read and stop batching once at least one complete line is buffered. Alternatively, gate the greedy batching so it doesn't apply when stdout is line-buffered / input is a tty-driven pipe. Add a test that feeds a reader delivering `"line\n"` then simulating "no more data yet" and asserts the line is surfaced without requiring buffer fill.
  - **Confidence:** 85/100

#### Medium

- **[test-gap]** No regression test for any of the changed behavior — only two existing constants were mutated — `crates/searcher/src/searcher/core.rs:198-205`, `crates/searcher/src/line_buffer.rs:419-425`
  - **What's wrong/missing:** The diff adds no new test function. Existing context tests only exercise `after_context`/`before_context` values of 1–3 (grep for `after_context([2+ digits]` returns nothing). The PR's actual target — **large** `-A` with `-B 0`, which is where the O(N²)→O(N) `roll()` retention change and the read-batching matter — has no test. The correctness of `lines::preceding(buf, term, before_context)` replacing `max_context()` rests entirely on tests that never span multiple buffers with a large context window. Likewise there is no test for the stdin read-amortization (the benchmark in the commit message used bulk file data, not a committed test).
  - **Why it matters:** The `roll()` change is subtle: it now retains only `before_context` preceding lines instead of `max_context()`. It appears correct (I traced that `last_line_visited` protects un-emitted after-context and the context separator), but a future change to `preceding`, `set_pos`, or retention accounting could silently drop after-context lines with large `-A`, and nothing would catch it. Operationally this is the exact regression class the PR is one refactor away from.
  - **Fix:** Add a searcher-level test with `before_context(0)`, a large `after_context` (e.g. 50–100), and a haystack that forces multiple `fill()`/`roll()` cycles (small heap limit), asserting byte-identical output to a slice search. Add a `LineBuffer`-level test that drives `fill()` with a reader returning short reads to lock in the batching contract.
  - **Confidence:** 88/100

#### Low

- **[docs]** `bytes_searched` / `--stats` "bytes searched" output silently changes for binary-quit inputs; undocumented — `crates/searcher/src/searcher/glue.rs:740,774`
  - **What's wrong/missing:** The two updated constants (`byte count:262146` → `262142`) reflect a real user-visible change: because the batched read scans more data before the binary-quit truncation, fewer bytes get counted as "searched." `byte_count` is surfaced publicly via `SinkFinish::byte_count()` (`sink.rs:348`), `--stats` "bytes searched" (`crates/core/main.rs:469`), and `--json` `bytes_searched` (`tests/json.rs:263`). The binary *offset* is unchanged, and the non-binary `json.rs` assertion (367) is unaffected, so nothing breaks — but users diffing `--stats`/`--json` across versions on binary corpora with `--binary`/quit detection will see a shifted count. The CHANGELOG advertises only a perf improvement, not any stats change.
  - **Why it matters:** Minor observability drift; could surprise anyone snapshotting stats for regression baselines. Low blast radius (narrow scenario: binary detection = quit, binary byte mid-buffer).
  - **Fix:** Either note the stats-value change in the CHANGELOG/PR, or confirm it's considered noise and leave as-is. No code change required.
  - **Confidence:** 78/100

### Most Critical Gap

The inner read loop in `fill()` no longer returns once a complete line is buffered — it blocks reading until the 64 KB buffer fills — which breaks prompt matching for streaming stdin, including the `tail -f … | rg … --line-buffered` pattern that ripgrep's own documentation advertises. Fix this before relying on the merge for any interactive/pipeline use, and add the missing large-`-A` and streaming-input regression tests.

### Positive Observations

- The `roll()` change is correct and well-reasoned: `consumed = max(context_start, last_line_visited)` guarantees un-emitted after-context and the context-separator position (via `last_line_visited`) survive even though fewer preceding lines are retained; the explanatory comment added at `core.rs:194-197` accurately documents the `before_context==0` reasoning.
- The author correctly identified and updated both dependent binary-detection test constants, and the binary *offset* invariant is preserved (only the searched-byte count shifts).
- The `Ok(0)`-on-first-read → EOF semantics are preserved from the old code; the batching does not introduce a premature-stop bug or infinite loop (I traced the roll/re-fill cycle after a mid-buffer EOF).
