## Findings

### #1 🟠 High: Two-step lookup still throws `AmbiguousMatchException` for a hide at an intermediate base

| | |
|---|---|
| **File** | `src/Components/Forms/src/EditContextDataAnnotationsExtensions.cs:376` |
| **Category** | correctness / spec-compliance (incomplete fix) |
| **Confidence** | 100 (validator: **confirmed**) |
| **Found by** | quick (High), broad (High), test-reviewer (High), spec-compliance (High), adversarial (High), dotnet (High/conf 50→corroborated), prior-feedback (High) — 7 finders |

**Issue:** The fix resolves the ambiguity only when the model's **exact runtime type** declares the hiding `new` property (step 1, `DeclaredOnly`). When the `new`-hide occurs at a **non-leaf ancestor** and the concrete model type does **not** redeclare the member, step 1 returns `null` and execution falls into the fallback:

```csharp
propertyInfo = cacheKey.ModelType.GetProperty(
    cacheKey.FieldName,
    BindingFlags.Public | BindingFlags.Instance | BindingFlags.FlattenHierarchy);
```

`BindingFlags.FlattenHierarchy` only affects inherited **static** members; with no `Static` flag it is a no-op, so this call is functionally identical to the original `GetProperty(name)` — an unrestricted hierarchy walk. For a shape like:

```csharp
class Base { public object Tag { get; set; } }
class Mid  : Base { public new string Tag { get; set; } }   // hides with a DIFFERENT signature
class Leaf : Mid  { }                                        // does NOT redeclare Tag
```

validating `Leaf.Tag` collects both `Mid.Tag` (string) and `Base.Tag` (object) — same name, different signatures — and `GetProperty` throws `System.Reflection.AmbiguousMatchException`, the exact exception the PR set out to eliminate. Hierarchy depth does **not** disambiguate when signatures differ (that is precisely why the original 2-level `int`-over-`object` case in issue #27095 threw). The throw is uncaught at the sole caller (`OnFieldChanged`, synchronous, no try/catch) and occurs **before** the cache write, so nothing is cached and every subsequent `NotifyFieldChanged` for that field re-throws — in Blazor Server this tears down the circuit on each keystroke.

This is exactly the still-open concern raised by reviewer **Youssef1313** in the PR's **unresolved** review thread ("this code can still throw if the previous call returned null and we get into here with some shadowing member in a base type"). It was merged without a code or test change addressing it. No test covers this shape: the only multi-level test (`ValidatesPropertyHiddenAtMultipleInheritanceLevels`) uses `DeepDerivedModel`, which itself redeclares `Tag` on the leaf, so step 1 resolves it and the fallback is never exercised (see Minor → Testing Gaps).

**Pre-existing nuance (validator-corrected):** The *exact* non-redeclaring-leaf shape would also have thrown under the original single-arg `GetProperty(name)` — this PR does not **regress** that shape. But the change's stated scope is to fix `AmbiguousMatchException` for hidden members generally (issue #27095), and it ships a fix plus a misleadingly-named "multi-level" test while leaving this same-class case live. It is therefore reported as a primary finding (incomplete fix), not filed under Pre-existing Issues. The adversarial-reviewer and validator both flagged the `pre_existing: true` nuance; 6 of 7 finders treated it as a defect of this change.

**Fix:** Make the fallback deterministic instead of re-running an ambiguous flattened lookup — walk the hierarchy one level at a time, `DeclaredOnly` at each level, taking the first (most-derived) match:

```csharp
propertyInfo = null;
for (var currentType = cacheKey.ModelType; currentType is not null; currentType = currentType.BaseType)
{
    propertyInfo = currentType.GetProperty(
        cacheKey.FieldName,
        BindingFlags.Public | BindingFlags.Instance | BindingFlags.DeclaredOnly);
    if (propertyInfo is not null)
    {
        break;
    }
}
```

Each level's `DeclaredOnly` lookup is unambiguous by construction, so no single call ever spans two hiding levels. Add a regression test for the non-redeclaring leaf shape above.

### #2 🟡 Medium: Dropping `BindingFlags.Static` is an undocumented behavior change; the two-step lookup carries no rationale

| | |
|---|---|
| **File** | `src/Components/Forms/src/EditContextDataAnnotationsExtensions.cs:370` |
| **Category** | knowledge-loss / behavior-change |
| **Confidence** | 100 (validator: **confirmed**) |
| **Found by** | dotnet (Medium), broad (Medium), knowledge (SHOULD→High), adversarial (Low) — dissenting severities; consolidated to Medium per the .NET specialist and validator |

**Issue:** The original `GetProperty(cacheKey.FieldName)` used the name-only overload whose default flags are `Public | Instance | Static`. Both new calls specify `Public | Instance` (no `Static`), so a public **static** property that previously matched a `FieldIdentifier` by name is now skipped (`TryGetValidatableProperty` returns `false` → the field is silently not validated, failing open). This is **intentional** — the new `IgnoresStaticProperty` test locks it in — and is arguably more correct (validating a static member through a per-instance `FieldIdentifier` is questionable). But nothing at the edit site records that the `Static` omission is deliberate, and the pre-existing comment above the lookup still only mentions "public properties." Separately, the whole two-step `DeclaredOnly`→fallback structure has **no comment** explaining *why* it exists (the `AmbiguousMatchException` workaround for `new`-hidden members, and why `DeclaredOnly` is tried first). A future maintainer could (a) "restore" `Static`, silently changing behavior a test now guards, or (b) collapse the two calls back to `GetProperty(name)`, reintroducing issue #27095 — the regression tests are the only backstop (which is why this is knowledge-loss, not a live bug).

**Dissent:** knowledge-reviewer rated the comprehension risk `SHOULD` (→High); adversarial rated the static fail-open `Low`. The .NET specialist (`dotnet-reviewer`) and the validator both settled on Medium.

**Fix:** Add a short comment above the lookup, e.g.:

```csharp
// A model may declare a `new` property that hides a same-named base property. The name-only
// GetProperty overload throws AmbiguousMatchException in that case, so resolve the most-derived
// declaration first (DeclaredOnly), then fall back to inherited properties. Static members are
// intentionally excluded — a FieldIdentifier is per-instance.
```

### #3 🟢 Low: `BindingFlags.FlattenHierarchy` in the fallback is inert and misleading

| | |
|---|---|
| **File** | `src/Components/Forms/src/EditContextDataAnnotationsExtensions.cs:378` |
| **Category** | comprehension-risk / dead flag (quotable fact) |
| **Confidence** | 100 (deterministic — documented reflection semantics; corroborated ×3, validation waived) |
| **Found by** | broad (Low), knowledge (COULD→Low), dotnet (Low) — 3 finders |

**Issue:** `BindingFlags.FlattenHierarchy` is documented to change lookup only for inherited **static** members. Combined with `Public | Instance` and no `Static`, it has no effect — inherited public **instance** properties are already returned without it. The hierarchy-spanning behavior of the fallback comes entirely from the **absence of `DeclaredOnly`**, not from `FlattenHierarchy`. The flag's presence implies it is load-bearing when it is not, which actively obscures the reasoning behind Finding #1 (a reader may believe the fallback does something safe that it does not).

**Fix:** Drop the inert flag — `BindingFlags.Public | BindingFlags.Instance` alone. (If the recommended per-level walk from Finding #1 is adopted, this call goes away entirely.)
