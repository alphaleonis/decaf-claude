### #1 🟠 High: `NewNodeConfig` omits `AddFunc` — initial-list replay is silently dropped

| | |
|---|---|
| **File** | `pkg/proxy/config/config.go:288` |
| **Category** | correctness / design |
| **Confidence** | 100 (validated: confirmed) |
| **Found by** | quick (High), broad (High), consistency (High), design (Medium), go (Medium) |

**Issue:** `NewNodeConfig` wires only `UpdateFunc`→`handleChangeNode` and `DeleteFunc` — no `AddFunc`. `NodeManager`'s node-scoped informer is `Start()`ed and cache-synced inside `newNodeManager` (during `newProxyServer`), while `nodeConfig.RegisterEventHandler(s.NodeManager)` runs later in `Run()`. client-go replays the current cache to a handler registered on an already-started informer as synthetic `Add` notifications; `ResourceEventHandlerFuncs.OnAdd` is a no-op when `AddFunc` is nil (`controller.go:257-261`), so that replay is dropped. The prior commit wired `AddFunc: result.handleAddNode`; patch 5 deleted it while only rewrapping `UpdateFunc`. Sibling `NodeTopologyConfig` (added in this same PR) and every other Config in the file wire `AddFunc`. The `NodeHandler` doc says `OnNodeChange` fires on "creation or modification."

**Consequence:** a NodeIP/PodCIDR/taint change that lands in the startup window (between `newNodeManager`'s poll and `Run()`'s registration — which spans health-server setup, `platformSetup`/conntrack/sysctls, and the initial proxier sync) is not acted on until the next genuine `Update` or the resync. Resync period is `ConfigSyncPeriod` (default 15 min), so the "crash immediately on NodeIP/PodCIDR change" guarantee is weakened for up to that long in that window. (Node *deletion* is unaffected — `DeleteFunc` is wired.)

**Fix:** wire `AddFunc: func(obj interface{}) { result.handleChangeNode(obj) }` alongside `UpdateFunc`, mirroring `NodeTopologyConfig`.

### #2 🟠 High: `klog.Flush()` replaces `klog.FlushAndExit()` — unbounded flush before `os.Exit`; exit-reason logs can be lost

| | |
|---|---|
| **File** | `pkg/proxy/node.go:154` (also `:170`, `:178`) |
| **Category** | reliability / observability |
| **Confidence** | 100 (validated: confirmed) |
| **Found by** | quick (High), knowledge (Low), adversarial (Medium), go (High), prior-feedback (High) |

**Issue:** the removed `NodePodCIDRHandler` exited via `klog.FlushAndExit(klog.ExitFlushTimeout, 1)` — a **bounded** flush (klog runs `Flush()` in a goroutine racing a timeout, precisely because flush hooks "may deadlock ... or take too long") followed by a guaranteed exit. All three `NodeManager` exit sites now do `klog.Flush(); n.exitFunc(1)`. Plain `klog.Flush()` (`klog.go:452`) is synchronous with no timeout, so if the log sink stalls, `exitFunc(1)`/`os.Exit` is never reached and the crash-restart safety net silently fails to fire. A prior reviewer (nojnhuh) reported a real production symptom: kube-proxy dying after "Caches are synced" with no crash-reason log, attributed to this change (verified in the PR's GitHub thread).

**Fix:** use a **bounded** flush before exiting — but note `klog.FlushAndExit` always calls the package-global `klog.OsExit`, bypassing the injectable `exitFunc` the tests rely on. So the correct fix flushes with a timeout (mirroring klog's `timeoutFlush`) and *then* calls `n.exitFunc(1)`, rather than substituting `FlushAndExit` verbatim.

### #3 🟠 High: `OnNodeChange` overwrites the stored node *before* validating its IPs → spurious `os.Exit` on IP loss→restore

| | |
|---|---|
| **File** | `pkg/proxy/node.go:145` |
| **Category** | correctness / reliability |
| **Confidence** | 100 (probe-confirmed + validated: confirmed) |
| **Found by** | adversarial (High) |

**Issue:** `OnNodeChange` locks, computes `oldNodeIPs` from the current `n.node`, then unconditionally does `n.node = node` and unlocks — **before** validating that `node` has usable IPs. If a transient update carries no addresses, `GetNodeHostIPs(node)` errors, the code logs and early-returns without exiting — but `n.node` has already been poisoned with the IP-less node. A subsequent update that **restores the original IP** then computes `oldNodeIPs` from the poisoned (empty) baseline, sees a "change," and calls `exitFunc(1)` — crashing kube-proxy even though the net NodeIP never changed (X → ∅ → X).

**Probe (run by the orchestrator, untracked package-internal test):** `event1` = update with no addresses → no exit (early return); `event2` = restore `192.168.1.10` → **`exit(1)` fired**. Bug reproduced; tree restored byte-identical afterward.

**Trigger realism (from validation):** in-tree kubelet/cloud-provider address setters are defensive and won't drop a running node's addresses to empty, so this needs an atypical actor — a less-defensive out-of-tree cloud provider, a stale full-object replace by a third-party controller/webhook, or an SSA field-manager conflict. Narrow, but the code explicitly anticipates the `GetNodeHostIPs` error (it handles it gracefully), so the state-corruption-on-error path is a real defect, not a can't-happen.

**Fix:** validate IPs on the incoming `node` *before* overwriting `n.node`; on `GetNodeHostIPs` error, log and keep the last-known-good baseline (do not store the IP-less node). The same store-before-validate ordering applies to the PodCIDR branch.

### #4 🟠 High: `TestNewNodeTopologyConfig` is tautological — it cannot detect removal of the dedup logic it exists to test

| | |
|---|---|
| **File** | `pkg/proxy/config/config_test.go:464` |
| **Category** | test-quality / false-positive |
| **Confidence** | 100 (probe-confirmed + validated: confirmed) |
| **Found by** | test (Critical) |

**Issue:** `nodeTopologyHandlerMock` records only the last received value (`topologyLabels`), never an invocation count. The "should receive no notification" steps assert only on the resulting value (`require.Empty`/`require.Len`). Production's dedup guard `if reflect.DeepEqual(n.topologyLabels, topologyLabels) { return }` suppresses redundant `OnTopologyChange` calls — but in every "no notification" step the computed labels equal what the handler already holds, so even if the guard were deleted (handler fires on every event with the same value), the asserted value is unchanged and every assertion still passes. The `callback()` driving `waitForInvocation` fires unconditionally (even on the dedup early-return), so it gives no signal either.

**Probe (run by the orchestrator):** the dedup guard in `config.go` was neutralized and `TestNewNodeTopologyConfig` **still passed**, with the handler observably firing on every event. Restored byte-identical afterward.

**Why it matters:** the dedup guard is the entire point of `NodeTopologyConfig` (avoid redundant, cluster-wide proxier re-syncs on every node event/resync). A regression there is a silent performance regression that **no other test would catch** — this is the only test exercising `handleNodeEvent`'s dedup path.

**Fix:** add an invocation counter to the mock and assert the expected call count at each step (especially the "no notification expected" steps).

### #5 🟡 Medium: Node deletion now crashes kube-proxy — intentional, but undocumented in code and with the design breadcrumb removed

| | |
|---|---|
| **File** | `pkg/proxy/node.go:176` |
| **Category** | reliability / knowledge |
| **Confidence** | 75 (validated: confirmed; **downgraded** from High) |
| **Found by** | broad (Critical), knowledge (High), design (Medium), adversarial (High) |

**Issue:** pre-PR, a Node-delete event was non-fatal — `NodeEligibleHandler.OnNodeDelete` marked the node ineligible (healthz → 503, graceful LB drain) while kube-proxy kept running. `NodeManager.OnNodeDelete` now unconditionally `exitFunc(1)` (os.Exit). Two real gaps remain: (1) the `NodeManager` doc comment says it "crashes kube-proxy if there are any changes in NodeIPs or PodCIDRs" — it does **not** mention that deletion is also fatal; (2) the `// https://issues.k8s.io/111321` design breadcrumb was removed from the tree.

**Downgrade rationale (from validation):** the crash-on-delete is a **deliberate, code-owner-endorsed** decision (danwinship in the PR thread: "we have basically normalized the fact that kube-proxy will exit and expect to be restarted ... treat the lack of a Node object as a fatal error and exit"). So this is not a correctness defect and the "restore graceful behavior" fix would run counter to maintainer intent. What remains is a **knowledge-preservation** gap plus an availability posture worth stating: a transient node-object delete→recreate (autoscaler, node recycling, etcd restore, mass `kubectl delete node`) now crash-loops kube-proxy where it previously degraded.

**Fix:** document in code that node deletion is intentionally fatal (and why), and restore the issue-111321 reference next to the PodCIDR-exit rationale so the design context isn't lost to `git blame`.

### #6 🟡 Medium: `NodeEligible()` deep-copies the whole Node and takes a now-vestigial exclusive lock on every `/healthz` request

| | |
|---|---|
| **File** | `pkg/proxy/healthcheck/proxy_health.go:176` |
| **Category** | performance / design |
| **Confidence** | 75 (validated: confirmed) |
| **Found by** | performance (High), broad (Medium), design (Low), quick (Low) |

**Issue:** `NodeEligible()` now calls `hs.nodeManager.Node()`, which returns `n.node.DeepCopy()` (a full `v1.Node` copy — `Status.Images`, `ManagedFields`, `Annotations`, `Conditions`, …), then reads only `DeletionTimestamp` and `Spec.Taints`. **Fix:** add a narrow `NodeManager` accessor (e.g. `IsEligible()` / `Taints()`) that reads the two fields under `NodeManager.mu` without a full copy, and drop the vestigial `hs.lock` from `NodeEligible()`.

### #11 🟢 Low: Fragile `if pollErr != nil { return nil, err }` contract in `newNodeManager`

| | |
|---|---|
| **File** | `pkg/proxy/node.go:107` |
| **Category** | maintainability / knowledge |
| **Confidence** | 75 |
| **Found by** | knowledge (Medium), prior-feedback (Low) |

**Issue:** on poll timeout the function returns the inner `err`, not `pollErr`. This is only safe because every non-success return in the closure sets `err` first; a future closure branch that returns `(false, nil)` without setting `err` would make `newNodeManager` return `(nil, nil)`, and `newProxyServer` (`if err != nil` is false) would then nil-deref `s.NodeManager.NodeIPs()`. **The correctness risk is not currently reachable** — three reviewers (quick, go, adversarial) independently confirmed via the vendored `wait` source that `immediate=true` runs the closure at least once and every failure path sets `err`. What remains is a fragile implicit invariant plus an **unaddressed prior-review request** (danwinship called the pattern "weird" and asked to simplify).

**Fix:** either document the "every non-success path must set `err`" invariant at the closure, or make it robust: `if pollErr != nil { if err == nil { err = pollErr }; return nil, err }`.
