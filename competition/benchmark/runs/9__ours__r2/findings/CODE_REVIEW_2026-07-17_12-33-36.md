### #1 🟠 High: Node handlers are registered on an already-started informer → data race on `eventHandlers` + possibly-lost initial topology label

| | |
|---|---|
| **File** | `pkg/proxy/config/config.go:485-511` (+ `cmd/kube-proxy/app/server.go:606-611`; root cause `pkg/proxy/node.go:76`) |
| **Category** | race-condition / concurrency |
| **Confidence** | 100 (mechanism) · 75 (practical manifestation) |
| **Found by** | broad-reviewer (Critical) — validated CONFIRMED |

**Issue:** `NewNodeManager` starts **and** cache-syncs its node informer inside the constructor (`node.go:76-79`), which runs early in `newProxyServer` (`server.go:211`). Later, in `Run()`, `NewNodeConfig(...)`/`NewNodeTopologyConfig(...)` attach event handlers to that **already-started** informer, each immediately followed by `RegisterEventHandler(...)`. This violates the register-before-start invariant the same file documents at `server.go:579-581`/`601-602` and honors for every other informer.

Attaching a handler to an already-synced shared informer makes client-go replay the current cache as a synthetic **Add**, delivered **asynchronously** by a newly-spawned listener goroutine (confirmed against vendored `client-go/tools/cache/shared_informer.go`). For `NodeTopologyConfig`, `AddFunc` is wired → the listener goroutine runs `handleNodeEvent` (which ranges `n.eventHandlers` and mutates `n.topologyLabels`, both **unlocked**) concurrently with the main goroutine's `RegisterEventHandler` append (also unlocked). That is a genuine data race on the `eventHandlers` slice (`NodeTopologyConfig` has no mutex, unlike its sibling `ServiceCIDRConfig`).

Worst case: if the synthetic-Add goroutine wins the race, it ranges an empty `eventHandlers` (proxier not yet registered) → the proxier never receives the initial `OnTopologyChange`. Because `handleNodeEvent` still sets `n.topologyLabels`, the `reflect.DeepEqual` guard (`:528`) suppresses the next identical resync, and there is **no `nodeTopologyConfig.Run()`** to re-deliver — so topology-aware routing for this node can be silently disabled until the zone label actually changes.

**Fix:** Don't pre-start `NodeManager`'s informer for event dispatch. Start it as part of `Run()`'s informer-start sequence **after** handlers are registered (as every other proxy informer does), or pass the handler(s) into the constructors, or guard `eventHandlers`/`topologyLabels` with a mutex. This one root cause also fixes #2.

*Severity note:* the finder rated this Critical; the validator and I calibrate to **High** — a verified data race whose demonstrated worst case is a missed initial topology-routing label (self-heals on the next label change), not crash/corruption/security.

### #2 🟠 High: `NewNodeConfig` omits `AddFunc` → a node change during the startup window is undetected until the next resync

| | |
|---|---|
| **File** | `pkg/proxy/config/config.go:288` (interface doc `:262-265`) |
| **Category** | correctness / data-loss / contract |
| **Confidence** | 100 |
| **Found by** | quick (High), broad (High), knowledge (High), design (Low), adversarial (High) — validated CONFIRMED |

**Issue:** `NewNodeConfig` registers only `UpdateFunc` and `DeleteFunc` — no `AddFunc` — while the sibling `NewNodeTopologyConfig` (same file, `:487-492`) does wire `AddFunc`. Because the informer is already synced when `NodeConfig` registers (see #1), client-go's late-join replay of the current node is delivered as a synthetic **Add**, which with a nil `AddFunc` is a no-op — silently dropped. `NodeManager` captured its baseline via the startup poll, so the *initial* state isn't lost; but any NodeIP/PodCIDR change that lands in the window between the poll (early in `newProxyServer`) and `RegisterEventHandler` (later in `Run()`, after proxier/backend setup) is not delivered to `OnNodeChange`, so the crash-on-change safety check does not fire for it until the next real `Update` or resync (default `ConfigSyncPeriod`). The `NodeHandler.OnNodeChange` doc ("called whenever **creation** or modification … is observed") also contradicts the implementation.

**Fix:**
```go
handlerRegistration, _ := nodeInformer.Informer().AddEventHandlerWithResyncPeriod(
    cache.ResourceEventHandlerFuncs{
        AddFunc:    result.handleChangeNode,          // harmless for NodeManager (compares, finds no diff)
        UpdateFunc: func(_, newObj interface{}) { result.handleChangeNode(newObj) },
        DeleteFunc: result.handleDeleteNode,
    },
    resyncPeriod,
)
```
Or fix the root cause per #1 (register handlers before starting the informer) and correct the interface doc.

### #3 🟠 High: `klog.Flush()` + `os.Exit` replaces `klog.FlushAndExit()` on the crash paths → exit-reason logs can be lost (open, unaddressed regression report)

| | |
|---|---|
| **File** | `pkg/proxy/node.go:154-155` (also `:170-171`, `:178-179`) |
| **Category** | reliability / observability / prior-feedback |
| **Confidence** | 100 |
| **Found by** | prior-feedback (High), go (High), adversarial (Medium) — validated CONFIRMED |

**Issue:** All three `NodeManager` exit sites do `klog.Flush(); n.exitFunc(1)` (production `exitFunc = os.Exit`). Plain `klog.Flush()` is unbounded; klog's `FlushAndExit` runs the flush in a goroutine bounded by a timeout **specifically because the hooks Flush invokes can deadlock** (klog's own `exit.go` comment). So a blocked flush can either lose the crash-reason line before `os.Exit` or never reach `os.Exit`. The PodCIDR-changed site (`:154`) directly replaced `klog.FlushAndExit(klog.ExitFlushTimeout, 1)`; the NodeIPs-changed (`:170`) and delete (`:178`) sites are new exit paths that adopt the same weaker pattern rather than the codebase's established bounded one.

An **unresolved** review thread (nojnhuh, filed after merge, no reply) reports kube-proxy exiting during cluster creation with only three log lines and no exit reason, and asks whether reverting to `FlushAndExit` (via overriding `klog.OsExit` in tests) would fix it.

> **[Unverified]** nojnhuh's causation ("this PR broke cluster creation") is the reporter's own speculation and is not established here. What *is* verified: (a) the code uses unbounded `klog.Flush()` + `os.Exit` instead of the bounded `FlushAndExit`, and (b) an open, unaddressed reliability report matches that change. This is expected-to-be-risky behavior, not a guaranteed failure.

**Fix:** Use `klog.FlushAndExit(klog.ExitFlushTimeout, 1)` (routed through the injectable `exitFunc`/`klog.OsExit` for tests) at all three sites so the exit reason is flushed under a bounded timeout.

### #6 🟡 Medium: `NodeEligible()` takes a vestigial exclusive lock and deep-copies the whole Node on every `/healthz` request

| | |
|---|---|
| **File** | `pkg/proxy/healthcheck/proxy_health.go:176-190` |
| **Category** | performance / concurrency-design |
| **Confidence** | 75 |
| **Found by** | broad (Medium), design (Low) — validated CONFIRMED |

**Issue:** `NodeEligible()` acquires `hs.lock.Lock()` (exclusive) but reads/writes **no** field guarded by `hs.lock` — it only reads the immutable `hs.nodeManager` pointer and calls `nodeManager.Node()`, which is independently synchronized and returns a full `DeepCopy()`. **Fix:** Drop the `hs.lock` acquisition entirely (node consistency is already provided by `NodeManager.Node()`); optionally avoid the deep copy by having `NodeManager` expose a narrow eligibility read.

### #7 🟡 Medium: `OnNodeChange` overwrites the cached node *before* validating its IPs → a transient address-less event poisons the baseline and causes a spurious crash

| | |
|---|---|
| **File** | `pkg/proxy/node.go:145` (with `:159-163`) |
| **Category** | error-handling / edge-case |
| **Confidence** | 75 |
| **Found by** | adversarial (Medium) — validated CONFIRMED |

**Issue:** `n.node = node` is assigned (`:145`) before the `GetNodeHostIPs(node)` error check (`:159-163`), and the early `return` on error never reverts it. ### #8 🟡 Medium: `NodeManager`'s informer is `Start(wait.NeverStop)`-ed before sync → goroutine/watch leak on the error-return paths and the informer ignores the passed `ctx`

| | |
|---|---|
| **File** | `pkg/proxy/node.go:76` |
| **Category** | goroutine-leak / context-propagation |
| **Confidence** | 75 |
| **Found by** | go-reviewer (Medium) — validated CONFIRMED |

**Issue:** The factory is started with `wait.NeverStop` (not `ctx`), and neither error return (`:77-79` sync failure, `:107-109` poll failure) stops it; the factory is a local never stored on the struct, so it can never be stopped even on success. ### #9 🟡 Medium: `OnNodeDelete` now unconditionally crashes kube-proxy — combined with the hard-fail 5-minute startup poll, a transient delete/recreate can drive CrashLoopBackOff

| | |
|---|---|
| **File** | `pkg/proxy/node.go:176-180` (cascade with startup poll `:87-109`) |
| **Category** | resilience / cascade |
| **Confidence** | 75 |
| **Found by** | adversarial (High) — validated CONFIRMED (severity → Medium) |

**Issue:** `OnNodeDelete` unconditionally calls `exitFunc(1)`. Previously node deletion did not crash kube-proxy (old handlers only logged / flipped health eligibility). Exiting on delete is **reviewer-intended** (danwinship: "by the end of the PR we should be exiting in this case too probably"), so this is not "it shouldn't exit." The finding is that the exit is unconditional with no grace for a transient delete/recreate (kubectl delete + kubelet re-register, etcd restore to an older revision, cloud node replacement reusing the name), and it cascades with the new hard-fail startup: on restart, `newNodeManager` blocks up to 5 min for the node to reappear with IPs and, if it doesn't, exits non-zero → CrashLoopBackOff, during which the node runs on stale/absent rules.

**Fix (optional / product decision):** consider a short grace/re-poll on delete before exiting, or mark the node ineligible and keep serving existing rules rather than exit immediately.

### #10 🟡 Medium: `NodeTopologyConfig`'s no-op `DeleteFunc` (doc claims it handles Delete) silently depends on `NodeManager` crashing on node delete

| | |
|---|---|
| **File** | `pkg/proxy/config/config.go:499` (doc `:513-514`) |
| **Category** | knowledge / consistency |
| **Confidence** | 75 |
| **Found by** | knowledge (Medium), consistency (Medium) |

**Issue:** `handleNodeEvent`'s doc says it handles "Add, Update and **Delete** events" (copying the sibling `handleServiceCIDREvent` convention), but the registered `DeleteFunc` is `func(_ interface{}) {}` — a no-op that never clears `topologyLabels`. This is only safe because `NodeManager` (registered on the **same** informer via `NodeConfig`) calls `os.Exit(1)` on delete, killing the process before stale labels matter. **Fix:** Reword the doc to "Add and Update," and add a comment on the empty `DeleteFunc` stating the no-op is safe only because `NodeManager` exits the process on node deletion (and must clear `topologyLabels` if that ever changes).
