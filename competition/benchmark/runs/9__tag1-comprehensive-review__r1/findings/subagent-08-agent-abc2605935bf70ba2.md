### Pass 2: Validated Findings

#### High

- **[Missing else/default — dropped initial-state handler]** `NodeConfig`'s node-informer registration omits `AddFunc`, silently dropping the synthetic initial-list replay that `client-go` delivers when a handler is added to an already-started/synced informer — `pkg/proxy/config/config.go:288-292`
  - **Unhandled path:** `NewNodeManager`/`newNodeManager` (`pkg/proxy/node.go:82-88`) now starts and syncs the node informer *before* `NodeConfig`'s handler is registered (inverted from the pre-PR ordering, which deliberately started the informer *after* registration — see the removed comment "This has to start after the calls to NewNodeConfig..."). Per `vendor`/`staging` `client-go` `shared_informer.go:697-721` (`AddEventHandlerWithOptions`), registering a handler on an already-started informer takes the "late join" path: it lists the indexer and delivers each current object as a synthetic `addNotification{isInInitialList: true}` — i.e., as an `OnAdd` call. `NodeConfig`'s registered `cache.ResourceEventHandlerFuncs{UpdateFunc: ..., DeleteFunc: ...}` has `AddFunc == nil`, and `ResourceEventHandlerFuncs.OnAdd` is a no-op when `AddFunc` is nil (`staging/.../cache/controller.go:257-260`). So this replayed event — the only mechanism carrying the Node's state as of `nodeConfig.Run()`'s handler registration to `NodeManager` — is silently discarded. Every other config type in the same file (`EndpointSliceConfig`, `ServiceConfig`, `ServiceCIDRConfig`, and the new `NodeTopologyConfig` added in this same PR at `config.go:487`) registers `AddFunc`; `NodeConfig` is the sole exception, indicating an oversight rather than intent.
  - **Consequence:** `NodeManager.node` is otherwise initialized once via `newNodeManager`'s poll (`nodeLister.Get`) at construction time. If the Node object's `Status.Addresses` or `Spec.PodCIDRs` change on the API server during the (non-trivial, multi-step) window between that poll succeeding and `Run()` registering `NodeConfig`'s handler — precisely the startup race `NodePodCIDRHandler`/`https://issues.k8s.io/111321` was originally built to guard against — `NodeManager` never observes that specific transition. `s.podCIDRs`/`s.NodeIPs` (captured once from the stale `NodeManager` state in `cmd/kube-proxy/app/server.go:217-218`) are baked into the proxier for the rest of its life with the wrong values, and kube-proxy's crash-and-restart self-healing mechanism does not fire for that drift. Detection is at best delayed until the next periodic resync (bounded by `ConfigSyncPeriod`, and not guaranteed at all if that period is configured to 0) delivers the *current* state as a genuine `Update`, at which point `OnNodeChange` finally diffs it against the stale baseline and exits — meaning kube-proxy runs with incorrect PodCIDR/NodeIP-based routing for a real but bounded window, rather than failing fast as designed.
  - **Remediation:** Add `AddFunc: func(obj interface{}) { result.handleChangeNode(obj) }` alongside `UpdateFunc` in `NewNodeConfig`'s `cache.ResourceEventHandlerFuncs` (mirroring the sibling `NodeTopologyConfig` registration a few lines below), so the late-join replay is delivered to `NodeManager.OnNodeChange` the same way it already is for `NodeTopologyConfig`.
  - **Confidence:** 78/100

### Positive Observations

- `NodeTopologyConfig`'s registration (`config.go:485-500`) correctly wires `AddFunc` to `handleNodeEvent`, so it *does* receive the late-join replay on the same shared informer — confirming the omission in `NodeConfig` is isolated rather than systemic.
- `handleDeleteNode`/`handleChangeNode` both correctly guard against unexpected object types and `cache.DeletedFinalStateUnknown` tombstones before dereferencing.
- `NodeManager`'s mutex correctly brackets reads/writes of `n.node` in `OnNodeChange`/`NodeIPs`/`PodCIDRs`/`Node`, and `Node()` deep-copies before returning, avoiding external aliasing of the cached object.
- The `NewNodeManager` poll loop correctly treats "not found" / "no IPs yet" / "no PodCIDR yet" as soft, retry-worthy conditions (`return false, nil`) rather than aborting the poll early, while still surfacing a descriptive error once the timeout is reached.
