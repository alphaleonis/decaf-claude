**Finding 1 — MidLevelModelWithShadow's shadow is dead weight; the test named for "multiple inheritance levels" never exercises it.**
`src/Components/Forms/test/EditContextDataAnnotationsExtensionsTest.cs:418-421` — `MidLevelModelWithShadow : ModelWithHiddenBaseProperty { public new string Tag { get; set; } }`. Severity: **Medium**.

- Confirmed via `grep -n "MidLevelModelWithShadow" src/Components/Forms/test/EditContextDataAnnotationsExtensionsTest.cs`: the only two occurrences in the whole file are the class declaration (line 418) and its use as `DeepDerivedModel`'s base type (line 423). It is never directly instantiated (`new MidLevelModelWithShadow(...)` does not appear anywhere).
- The only test that walks this hierarchy, `ValidatesPropertyHiddenAtMultipleInheritanceLevels` (`test/EditContextDataAnnotationsExtensionsTest.cs:241-254`), constructs a `DeepDerivedModel`, whose own runtime type re-declares `Tag` directly at `test/EditContextDataAnnotationsExtensionsTest.cs:425-426` (`[Range] public new int Tag`).
- The production fix's lookup order (`src/Components/Forms/src/EditContextDataAnnotationsExtensions.cs:370-372`) tries `cacheKey.ModelType.GetProperty(name, Public|Instance|DeclaredOnly)` first, against the model's *runtime* type. Because `DeepDerivedModel` declares `Tag` on itself, this first (`DeclaredOnly`) lookup always succeeds, and the `FlattenHierarchy` fallback at lines 376-378 is never reached for this test.
- Net effect: `MidLevelModelWithShadow.Tag` — the intermediate `new string Tag` shadow that the class name implies is the point of the fixture — is never the property resolved by either lookup branch in any test. It's present in the type graph purely as an inert link, not as something the test drives through. The test name "…AtMultipleInheritanceLevels" overpromises: it only proves `DeclaredOnly` still works when there happen to be extra ancestor levels underneath a self-redeclaring leaf, not that a shadow introduced at a *non-leaf* level is correctly resolved (which would require the `FlattenHierarchy` fallback path and is the scenario the class name suggests).
- Note also: no test in this diff exercises the `FlattenHierarchy` fallback (lines 376-378) against a genuinely ambiguous multi-level "new" chain at all. `ValidatesInheritedPropertyFromBaseClass` (`test/EditContextDataAnnotationsExtensionsTest.cs:225-238`) reaches that fallback but for `BaseName`, which is declared exactly once in its hierarchy (`ModelWithBaseName`, no hiding anywhere) — so it isn't a regression test for the ambiguous-hiding bug this PR title references, just for plain inheritance. This makes `MidLevelModelWithShadow` doubly moot: even if it were reachable, its presence is not what proves the fallback branch is ambiguity-safe.
- To actually exercise what the class's name and position imply, a test would need a fourth class extending `MidLevelModelWithShadow` *without* redeclaring `Tag`, so `DeclaredOnly` fails at the leaf and `FlattenHierarchy` must resolve a real multi-level "new" chain. As written, `MidLevelModelWithShadow` can be deleted (making `DeepDerivedModel` derive from `ModelWithHiddenBaseProperty` directly) with zero change in test behavior or coverage — it is unused in the sense the task description flagged.
