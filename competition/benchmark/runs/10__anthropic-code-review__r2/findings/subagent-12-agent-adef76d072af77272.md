## Considered But Not Flagged

**(Q1) `self.end += readlen` — OOB / panic via a hostile `Read` impl** — `line_buffer.rs:420-424`. `free_buffer()` returns `&mut self.buf[self.end..]`, so a contract-honoring `Read` returns `readlen <= free_buffer().len()` and `self.end` stays `<= self.buf.len()`. A buggy/hostile `Read` returning `readlen > buf.len()` would drive `self.end` past `self.buf.len()`, and the subsequent `&mut self.buf[oldend..self.end]` (line 431) or the next `self.free_buffer()` (line 419) would **panic** (bounds check) — never memory-unsafe, because `io::Read` is a safe trait operating only on the slice it was handed. Dismissed because: (a) the `self.end += readlen` accumulation existed identically **before** this change — the trust boundary is unchanged; the change only wraps the read in an inner loop, and every iteration re-derives `free_buffer()` so each read is still bounded by remaining space; (b) the readers in ripgrep's path are `std` `File`/`Stdin`(-like) streams that honor the contract — `readlen` is not attacker-controlled (the *bytes* are, the *count* is not); (c) worst case is a safe panic, not OOB/UB. Not exploitable, boundary unchanged.

**(Q2) Draining reads until the free buffer is full growing memory past the cap** — `line_buffer.rs:416-425`. The inner `while !self.free_buffer().is_empty()` loop only writes into **already-allocated** free space; it never calls `ensure_capacity()`. Allocation happens exclusively at the top of the outer `loop` (line 417), and `BufferAllocation::Error(limit)` (lines 513-518) remains the sole gate — `n == 0 → alloc_error`. So the inner loop fills the current buffer but cannot enlarge it. Peak `self.buf.len()` is identical to the pre-change design; the change is memory-neutral (same capacity, filled more eagerly per pass). `ensure_capacity()`/`BufferAllocation::Error` is still the only memory gate. No new memory-exhaustion surface.

**(Q3) Accumulated binary detection weakening "quit on binary"** — `line_buffer.rs:431-452`. This is the one real behavioral change: pre-change, binary detection ran after each single `read` (~64K granularity for `stdin`); post-change it runs once per **full buffer fill**, so on the fill where the binary byte first appears, up to the entire (possibly large, when `-A` has grown the buffer) free region is read and scanned *before* the `Quit` fires. Examined and dismissed as a defect because: (a) the **guarantee is preserved** — on `Quit`, `self.end = oldend + i` truncates to the binary byte and `buffer()` = `buf[pos..last_lineterm]` still excludes the binary byte and everything after it; bytes read past it are beyond `end` and never observable, and `binary_byte_offset` is recorded correctly; (b) the extra bytes land in **already-allocated** memory — bounded by the allocation cap, not additional heap; (c) it is the **intended** amortization behavior of the fix (fill the buffer, then process), and once `binary_byte_offset` is set the very next `fill()` early-returns at lines 410-412, so the "read more before quit" cost is a one-time, bounded latency/CPU effect on a single fill, not an escalating DoS. Reliability-relevant only as a minor, bounded latency increase on binary input combined with a large `-A` — not a security control gap.

**`newbytes.is_empty()` EOF path** (`line_buffer.rs:432-437`) and **partial-read / mid-fill EOF** — Traced: immediate 0-byte read ⇒ `newbytes` empty ⇒ `last_lineterm = end`, EOF return (equivalent to the old `readlen == 0` branch); a partial read followed by a 0-read breaks the inner loop with `end` advanced, so the trailing unterminated line is retained in `buffer()` and not dropped. Correct; no data loss.

**core.rs `roll()`: `preceding(buf, term, before_context)` vs `max_context()`** (`core.rs:198-202`) — Since `before_context <= max_context() = max(before, after)`, `preceding` searches for fewer (or equal) preceding lines, yielding a **larger-or-equal** `context_start`, hence `consumed = max(context_start, last_line_visited)` is larger-or-equal ⇒ **less** data retained across the roll. Memory-favorable and gated by the same `max_context() == 0` fast path. After-context is read forward, not retained from before the match, so keying retention on `before_context` is the correct quantity. No memory-growth or trust concern; functional correctness is covered by the passing suite and is outside security scope.

## Threat Model Notes

- **Trust boundary (unchanged):** Untrusted bytes enter via `rdr.read()` from `stdin`/files. The `Read` *implementation* (std `File`/`Stdin`) is trusted for its return-count contract; only the *byte content* is attacker-controlled. This boundary is identical before and after the change — the PR restructures read *scheduling*, not the trust model.
- **Memory-exhaustion posture:** Peak resident memory is governed entirely by `ensure_capacity()` + `BufferAllocation`. Default is `BufferAllocation::Eager` (documented "allocate without limit"), which under a single very long unterminated line will grow until OOM — this is **pre-existing, by-design, unchanged by this PR**, and mitigated in the ripgrep binary via its configurable heap/line limits (`BufferAllocation::Error`). The reviewed change neither introduces nor widens this; it does not touch `ensure_capacity()`.
- **Integer overflow:** `self.end += readlen`, `absolute_byte_offset += amt as u64`, and `binary_byte_offset = absolute_byte_offset + end` all operate on values bounded by `buf.len()` (≤ `isize::MAX`) for contract-honoring readers; no realistic overflow. `ensure_capacity`'s `limit - used` cannot underflow because `used` never exceeds `limit` (allocation is clamped to `limit - used` each step). Unchanged by the PR.
- **DoS relevance to #3184:** #3184 is a *performance* regression (super-linear slowdown on `stdin` with large `-A`); this change is its fix and is memory-neutral. It does not introduce a new DoS — the only new behavior (Q3: more bytes read per fill before binary detection) is bounded by the pre-existing allocation cap.
- **Net security assessment:** No systemic control was added, removed, or bypassed. Trust boundary unchanged, memory cap intact and still the sole gate, binary-quit "byte never observable" guarantee preserved, retention across rolls reduced. No architectural security gap attributable to this changeset.
