## Architectural Analysis

### Design Assessment

The PR extends the established `Canon/*.Canon.cs` add-on convention into a new variant: common code now *calls into* canon-specific logic, requiring paired `*.NonCanon.cs` stubs for projects without canonical-type support. The partial-class wiring itself is complete and self-enforcing at compile time. The significant architectural concern is that the fix widens the semantics of the widely-shared `CanCastTo` predicate from "cast definitely succeeds" to "cast is canon-plausibly compatible" for every caller, and at least one unmodified caller (`compareTypesForCast` in the JIT-EE interface) documents and relies on the old contract.

### Findings

#### Critical

- **[Coupling and Cohesion / Public API contract]** `CanCastTo` now returns canon-wildcard positives globally, violating a documented invariant in the JIT-EE interface — `src/coreclr/tools/Common/TypeSystem/Common/CastingHelper.cs:426`
  - Why it matters: The edits at `CastingHelper.cs:426,433,477,552` make `IFoo<__Canon>.CanCastTo(IFoo<string>)` return `true` (verified by tracing: `CanCastToNonVariantInterface` → `IsCanonEquivalent` → `IsCanonicalTypeArgMatch(__Canon, string)` → `string.IsGCPointer` → true). The unmodified `compareTypesForCast` at `src/coreclr/tools/Common/JitInterface/CorInfoImpl.cs:2946-2952` calls `fromType.CanCastTo(toType)` in the shared-from/exact-to branch and states "Pass back positive results unfiltered" → `TypeCompareState.Must` — while its own expected-result table at lines 2964-2965 says `IFoo<__Canon> -> IFoo<string>` must be `May`. Under the old `CanCastTo` this returned false and fell through to `May` (line 2972-2974); under the new one it returns `Must`. [Inference] A `Must` answer lets RyuJIT fold `isinst`/`castclass` to unconditional success in shared generic code where the runtime instantiation may not actually match — a silent miscompile risk affecting both NativeAOT and crossgen2 (both consume `ILCompiler.TypeSystem`, which now includes the Canon partials). The contract violation is verified from the code; the runtime miscompile consequence is inferred, not executed.
  - Strongest counter-argument: the author may have audited all `CanCastTo` callers and concluded canonical shapes never reach this branch in practice — but the branch at `CorInfoImpl.cs:2939` exists precisely to handle canonical `fromType`, and its comment was not updated, which suggests it was not audited. If evidence shows the branch is unreachable with these shapes, this downgrades to a stale-comment Medium.
  - Recommendation: scope the canon-wildcard behavior instead of globalizing it — either thread a mode flag through `CanCastToInternal` so constraint validation opts in (constraint checking is the only motivated caller), or fix `compareTypesForCast` to demote canon-involved positive results to `May`. Rejected alternative: reverting the `CastingHelper.cs` edits and reimplementing base-type/interface walking inside `TypeSystemConstraintsHelpers` — rejected because it duplicates the entire casting walker. Other callers inheriting the widened semantics without modification: `DevirtualizationManager.cs:97,133`, `MetadataVirtualMethodAlgorithm.cs:783-1167`, `ComparerIntrinsics.cs:225-235` — each should be audited under the new contract.
  - Confidence: 82/100

#### Medium

- **[Maintainability / incomplete symmetry]** `MakeGenericMethodSite` does not normalize its instantiated result while its mirror `MakeGenericTypeSite` now does — `src/coreclr/tools/aot/ILCompiler.Compiler/Compiler/Dataflow/HandleCallAction.cs:769`
  - `MakeGenericTypeSite` (line 785-788) gained `NormalizeInstantiation()` with a comment that `InstantiateSignature` "could end up with a denormalized shape (Foo<object, __Canon>)". `MakeGenericMethodSite` (line 769) uses the identical `InstantiateSignature` + `CheckConstraints` mechanism on the same runtime-determined instantiations and got no equivalent treatment (and no `MethodDesc` counterpart of `NormalizeInstantiation` exists — only the `TypeDesc` extension at `TypeExtensions.cs:678`). Either the method path has the same latent denormalization bug, or it is safe for a non-obvious reason that is undocumented.
  - Recommendation: add the equivalent normalization for the method path, or a comment on `MakeGenericMethodSite` explaining why denormalized shapes cannot occur / are harmless there. Rejected alternative: leaving it as-is on the assumption downstream canonicalization (e.g., canon method target resolution) absorbs it — rejected because that assumption is unverifiable from the diff and the asymmetry will confuse the next maintainer regardless.
  - Confidence: 78/100

- **[Maintainability / duplicated invariant]** The core canon-wildcard rule ("`__UniversalCanon` accepts anything; `__Canon` accepts GC pointers") is now encoded in three places — `src/coreclr/tools/Common/TypeSystem/Canon/TypeSystemConstraintsHelpers.Canon.cs:159`
  - The identical Universal→true / Specific→`IsGCPointer` clause pair appears in `IsCanonicalCastTarget` (`CastingHelper.Canon.cs:37-41`), twice symmetrically inside `IsCanonicalTypeArgMatch` (`CastingHelper.Canon.cs:54-64`), and in `CanCastToConstraintWithCanon` (`TypeSystemConstraintsHelpers.Canon.cs:159-162`). The rule is subtle enough that drift between copies would be a correctness bug, and the private-member/partial-class structure prevents the two classes from sharing it today.
  - Recommendation: extract a single internal helper (e.g., on `TypeSystemContext` in the Canon partial, next to `IsCanonicalDefinitionType`, which all three already call) and have all sites delegate to it. Rejected alternative: keep the copies for local readability — rejected because each copy already differs slightly in surrounding clauses, which is exactly the condition under which silent drift happens.
  - Confidence: 85/100

- **[Design Patterns / undocumented inclusion contract]** The `NonCanon` stub files carry no documentation of when a project must include them versus the Canon implementations — `src/coreclr/tools/Common/TypeSystem/Common/CastingHelper.NonCanon.cs:7`
  - Missing either partial fails compilation (undefined method) and including both fails (duplicate member), so gross wiring errors are caught. But including the *wrong* variant compiles cleanly with silently different casting behavior — a canon-supporting project wired to the NonCanon stubs would quietly revert to the pre-fix (buggy) semantics. Neither stub file (`CastingHelper.NonCanon.cs`, `TypeSystemConstraintsHelpers.NonCanon.cs`) has a comment stating "include only in projects that do not compile `TypeSystem\Canon`", and the choice is visible only in csproj wiring. The stub `TypeSystemConstraintsHelpers.NonCanon.cs:4` also has an unused `using System.Diagnostics;`.
  - Recommendation: add a header comment to both stub files stating the inclusion rule (pair with `*.cs`; mutually exclusive with the `Canon\` counterpart; choose based on whether the project compiles `TypeSystem\Canon\TypeSystemContext.Canon.cs`). Rejected alternative: an MSBuild-level consistency check — rejected as disproportionate for a two-file convention; a comment at the point of copy-paste is sufficient.
  - Confidence: 76/100

### Positive Observations

- The partial-class wiring is complete and consistent across all three consumers (verified by repo-wide grep — only three project files compile these sources): `System.Private.TypeLoader.csproj` compiles `CastingHelper.cs` + `CastingHelper.Canon.cs` and does not compile `TypeSystemConstraintsHelpers` at all (so no partial is needed there); `ILVerification.projitems` pairs both base files with both NonCanon stubs (correct — it includes no `TypeSystem\Canon` sources); `ILCompiler.TypeSystem.csproj` pairs both with both Canon implementations. No missing or mismatched inclusions.
- Giving `System.Private.TypeLoader` the Canon partial keeps runtime type-loader casting semantics aligned with compile-time analysis — the exact cross-component consistency this shared-source architecture exists to provide.
- The compile-time enforcement property of the stub pattern (exactly one partial per project or the build breaks) is a genuinely good choice over `#if` conditional compilation, which fails open.
- Constraint-validation test coverage (`ConstraintsValidationTest.cs`, +176 lines) systematically covers the wildcard matrix: special constraints, both canon kinds, both directions (canon as param and as constraint), arrays, nesting, and variance.

### Recommendations

1. Audit the unmodified `CanCastTo` callers — above all `compareTypesForCast` (`CorInfoImpl.cs:2915-2974`), whose inline result table now contradicts the predicate it calls — and either scope the canon-wildcard semantics to constraint validation or demote canon-involved positives to `May` at that call site.
2. Resolve the `MakeGenericMethodSite`/`MakeGenericTypeSite` normalization asymmetry (fix or document).
3. Consolidate the triplicated canon-wildcard rule into one helper next to `IsCanonicalDefinitionType`.
4. Document the Canon/NonCanon inclusion contract in the stub file headers.
