## Considered But Not Flagged

- **`if pollErr != nil { return nil, err }` in `newNodeManager` (`pkg/proxy/node.go:107`)** — verified this is correct, not a bug: the polled closure always returns `(false, nil)` on failure only after assigning a non-nil `err`, and returns `(true, nil)` only immediately after a successful `GetNodeHostIPs` call leaves `err == nil`. Since `wait.PollUntilContextCancel`'s own error is non-nil only due to context timeout/cancellation (the condition function never itself returns a non-nil error), `err` is guaranteed non-nil whenever `pollErr` is non-nil. Returning the descriptive `err` instead of the generic `pollErr` is intentional (per the adjacent comment) and matches the test expectations (`expectedError` assertions on specific messages like `"does not have any PodCIDR allocated"`).
- **Goroutine "leak" from `thisNodeInformerFactory.Start(wait.NeverStop)` when `newNodeManager` fails the poll** — real but matches a pre-existing codebase idiom (the old `currentNodeInformerFactory.Start(wait.NeverStop)` had the same unconditional lifetime), and in production a failed `newProxyServer` normally leads to process exit anyway. Only mildly relevant to unit tests (each failing `TestNewNodeManager` subtest leaks informer goroutines for the life of the test binary); not worth a hard finding.
- **`NodeManager.OnNodeChange` mutates `n.node` before deciding whether to crash** — there is a narrow window between `n.mu.Unlock()` and `n.exitFunc(1)` actually terminating the process where a concurrent `Node()`/`NodeIPs()` caller could observe the about-to-be-fatal new state. In production `exitFunc` is `os.Exit`, so the window is negligible; not worth flagging.
- **Shared `topologyLabels` map reference passed to multiple `OnTopologyChange` handlers** (`pkg/proxy/config/config.go`) — checked all consumers (iptables/ipvs/nftables/metaproxier/winkernel proxiers, `topology.go`); none mutate the map in place, only read from it, so no aliasing corruption risk.
- **`ProxyHealthServer`/`NodeManager` nil-pointer risk for HollowProxy** — `HollowProxy` never sets `NodeManager` or `HealthzServer`, and `Run()`'s `s.NodeManager != nil` guard plus `serveHealthz`'s nil handling prevent a crash; the real (non-hollow) path always constructs `NodeManager` before `HealthzServer`, so no reachable nil-deref.
