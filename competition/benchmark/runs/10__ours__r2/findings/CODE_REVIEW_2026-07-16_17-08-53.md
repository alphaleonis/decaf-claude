## Findings

### #1 🔴 Critical: New inner read-loop withholds already-read matching lines until the buffer fills or EOF — breaks `--line-buffered` / streaming `stdin`

| | |
|---|---|
| **File** | `crates/searcher/src/line_buffer.rs:419-425` |
| **Category** | correctness / reliability (silent output-withholding regression) |
| **Confidence** | 100 (runtime-reproduced + validator-confirmed + upstream-confirmed) |
| **Found by** | broad-reviewer (High), performance-reviewer (Medium), knowledge-reviewer (undocumented-tradeoff angle); validator: **confirmed**, recommended High |

**Issue:** Before this PR, `LineBuffer::fill` issued one `rdr.read()` per outer-loop iteration and returned `Ok(true)` as soon as the freshly-read bytes contained a line terminator. The new inner loop keeps reading until `free_buffer()` is exhausted (buffer physically full — 64 KB by default, larger when grown by a big `-A/-B/-C`) or `read()` returns `0` (true EOF), and only *then* runs binary detection / the line-terminator search / the early return:

```rust
let oldend = self.end;
while !self.free_buffer().is_empty() {
    let readlen = rdr.read(self.free_buffer())?;
    if readlen == 0 { break; }
    self.end += readlen;
}
// binary detection + rfind(lineterm) + early return happen only AFTER the loop
```

On a blocking producer that trickles data without closing the stream (`tail -f log | rg foo --line-buffered`, `journalctl -f | rg ...`), the second `read()` blocks waiting for more input, so an already-buffered complete matching line is **not** searched or emitted until ~64 KB accumulates or the pipe closes. `--line-buffered` cannot compensate: it controls *output* flush timing, but the block happens *before* a match is even found. ripgrep's own `--line-buffered` docs name `tail -f something.log | rg foo --line-buffered` as a supported use case, so this is in-scope, not a non-goal.

**Runtime reproduction (this review):** built the merged binary and a pre-fix variant (single read per outer iteration) and drove each with a producer that emits one match and then holds the pipe open for 6 s, under a 4 s timeout:

| Build | `(printf 'match1\n'; sleep 6) \| timeout 4 rg --line-buffered match` |
|-------|------|
| **merged (post-fix)** | *no output* — killed by timeout (match withheld) |
| **pre-fix (single read)** | `match1` emitted immediately |
| control (immediate EOF) | `match1` in 0.11 s |

**Upstream confirmation (verified via `gh`):** this exact regression shipped in ripgrep 15.0.0, was reported as issue **#3194** ("Line buffering appears broken in 15.0.0", CLOSED), and was fixed by follow-up **PR #3195** / commit `d47663b` ("searcher: fix regression with `--line-buffered` flag"). The `line_buffer.rs` half of this PR is what regressed; the independent `core.rs` `roll()` half is correct and not implicated (see Considered But Not Flagged).

**Fix:** Return to the caller as soon as a complete line is available rather than insisting on a full buffer — e.g. break out of the inner loop once `self.buf[oldend..self.end]` contains the line terminator (preserving read amortization for genuinely-large available reads while restoring streaming responsiveness). This is essentially what upstream did in #3195.

### #2 🟡 Medium: New inner read-loop has no in-code rationale — a future "simplification" silently reintroduces #3184

| | |
|---|---|
| **File** | `crates/searcher/src/line_buffer.rs:419-425` |
| **Category** | knowledge-preservation / comprehension-risk |
| **Confidence** | 75 |
| **Found by** | knowledge-reviewer (SHOULD); validator: **confirmed**, recommended Medium (downgraded from High) |

**Issue:** The outer `loop` already re-reads until a line terminator is found, so the added inner `while !self.free_buffer().is_empty()` loop reads as redundant to anyone who doesn't know its purpose: a single `read()` on `stdin` returns at most ~64 KB regardless of the destination size, which defeats amortization of the per-iteration work (`ensure_capacity`, binary detection, `rfind`) once a large `-A` grows the buffer well past 64 KB. That "why" lives only in the commit message; the `fill` doc comment (lines 389-405) was not updated and there is no comment on the loop. A maintainer collapsing the two nested loops back to a single read silently reintroduces the #3184 slowdown.

The validator downgraded from High to Medium because (a) `binary3`/`binary4` provide a partial byte-count regression guard (a careless full revert breaks CI — see the probe under Minor → Testing Gaps), and (b) exhaustive commit messages are this project's established "why" record. The fix is a cheap one-line comment and clearly worth doing.

> Note the tension with Finding #1: the *ideal* fix for #1 (return early once a terminator is present) is not a full revert, so any comment added here should explain the amortization intent without implying the whole-buffer fill is required for correctness.

**Fix:** Add a comment above the inner loop, e.g. *"Fill the free buffer as much as possible per outer iteration: a single `read` on stdin returns at most ~64K, which defeats amortization of the per-iteration work below (binary detection + line-terminator scan) when a large `-A/--after-context` grows the buffer."*

### Testing Gaps

- `crates/searcher/src/line_buffer.rs` / `glue.rs` (test modules) — no test uses a custom `io::Read` that returns genuine multi-chunk short reads, which is the exact scenario the new inner loop exists to handle; all `LineBufferReader` tests read from `&[u8]` (test-reviewer anchor 100, broad-reviewer anchor 75). **Probe result (orchestrator):** forcing one read per outer iteration made `binary3`/`binary4` **fail** (byte count `262142`→`262146`) while the other 75 tests passed. ### Considered But Not Flagged (all agents)

- **`core.rs` `roll()`: `max_context()` → `before_context` correctness** — independently verified correct by quick, broad, rust, and spec-compliance reviewers. `before_context ≤ max_context()`, so the new `context_start` from `preceding()` is ≥ the old one ⇒ `consumed` is never smaller ⇒ the change never *drops* buffer that the old code kept; `buf.len() - consumed` cannot underflow. After-context needs no backward retention across a roll — it is emitted forward via the persistent `after_context_left` counter, and when after-context is pending `last_line_visited` dominates `max(context_start, last_line_visited)`. No after-context bytes are dropped at a buffer boundary. This half of the PR is correct and is *not* implicated in Finding #1.
- **`self.end += readlen` / `self.buf[oldend..self.end]` overflow or slice panic** — safe (quick, broad, rust): `read()` is called with `self.free_buffer()` (`&mut self.buf[self.end..]`), so per the `Read` contract `readlen ≤` remaining capacity and `self.end` can never exceed `self.buf.len()`.
- **EOF path (`newbytes.is_empty()`)** — behavior-equivalent to the old `readlen == 0` early return; at most one extra no-op read syscall at true EOF (quick, broad, rust).
- **`BinaryDetection::Convert`/`Quit` offset correctness** — reported `binary_byte_offset` and truncation point unchanged; `find_byte`/`replace_bytes` use first-occurrence semantics over the same bytes whether scanned in one merged span or piecewise (broad, spec-compliance). *Quit-mode does more wasted I/O before detecting — same root cause as Finding #1, folded into it.*
- **Removing `.as_bytes_mut()`** — a no-op bstr identity conversion on `&mut [u8]`; its removal aligns the call with every other `.read()` site in the workspace (consistency, rust).
- **`ErrorKind::Interrupted`/`WouldBlock`** — propagated via `?` identically to the pre-diff code; not a new risk (rust, consistency).
