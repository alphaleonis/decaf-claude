## Considered But Not Flagged

- **`self.end += readlen` / `self.buf[oldend..self.end]` overflow (task item a)** — `Read::read`'s documented contract guarantees `n <= buf.len()` for the slice passed in. Since `rdr.read(self.free_buffer())` is called with `self.free_buffer()` (`&mut self.buf[self.end..]`) as the buffer, a compliant reader can never return more than the remaining capacity, so `self.end` cannot exceed `self.buf.len()` and the later slice cannot panic. This reliance is identical to the pre-diff single-read version (`self.end += readlen` was already unconditional there) — not a new risk introduced by this change. (confidence 25 as a *novel* finding — pre-existing, universal Rust idiom of trusting the `Read` contract; not reported.)

- **`readlen == 0` as EOF signal (task item b)** — Correct. The inner `while !self.free_buffer().is_empty()` guard ensures `read()` is only ever called with a non-empty buffer, so per the `Read` trait docs a `0` return in that call can only mean genuine (possibly temporary) EOF, not "buffer was zero-length." No ambiguity is introduced.

- **`ErrorKind::Interrupted`/`WouldBlock` handling** — Both old and new code propagate errors via `?` without special-casing `Interrupted`. Behavior is unchanged from the pre-diff version; not a regression introduced by this change.

- **Removing `.as_bytes_mut()` (task item c)** — `free_buffer()` already returns `&mut [u8]`; `.as_bytes_mut()` was a no-op bstr extension call. No type or mutability change, no new aliasing.

- **Repeated `self.free_buffer()` calls in the `while` condition and body (task item d)** — Each call produces an independent short-lived `&mut self.buf[..]` borrow that ends immediately after use (NLL). No borrow held across loop iterations, no compile-time or runtime aliasing hazard — this is exactly the kind of thing the borrow checker already guarantees, so it's out of scope even if it looked suspicious at first glance.

- **Streaming/interactive-stdin latency** — The inner loop now fills the *entire* free buffer (or blocks until real EOF) before running line-terminator search, rather than searching after every single `read()` call as the old code did. For a slow-trickling producer (e.g., an interactive terminal or `tail -f`-style pipe) this delays output until the buffer fills or the stream closes — a genuine behavior change. However, this is precisely the documented, intentional trade-off from the upstream commit message (`8c6595c`/`d4b77a8`, benchmarked and accepted by the maintainer to fix the exponential-slowdown bug). It's a design/behavioral trade-off, not a Rust-semantics defect (no panic/unsafe/lock/async hazard), so it belongs to design/performance review scope, not this one.

- **`Core::roll` using `before_context` instead of `max_context()`** — Verified `before_context <= max_context()` always (`max_context = max(before_context, after_context)`), so the new `context_start` from `lines::preceding` is `>=` what the old call would have produced, meaning `consumed <= buf.len()` still holds and `buf.len() - consumed` in `set_pos` cannot underflow. This is a pure algorithmic correctness change (matching context-window semantics to actual need), not a Rust-idiom issue — no new arithmetic/slice panic surface.
