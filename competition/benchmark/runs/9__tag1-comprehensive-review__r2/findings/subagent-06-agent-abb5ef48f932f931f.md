## Summary

This PR consolidates `OnNodeAdd`/`OnNodeUpdate` into `OnNodeChange` and splits node-eligibility (`NodeManager`, crash-on-IP/PodCIDR-change) from topology-label propagation (`NodeTopologyConfig`/`OnTopologyChange`). The new `pkg/proxy/node_test.go` (`TestNewNodeManager`, `TestNodeManagerOnNodeChange`, `TestNodeManagerOnNodeDelete`, `TestNodeManagerNode`) and `pkg/proxy/config/config_test.go` (`TestNewNodeTopologyConfig`) are genuinely more thorough than what they replace at the **unit level** for the individual handler methods: initial sync/retry, NodeIP-change-triggers-exit, PodCIDR-change-triggers-exit (gated), delete-triggers-exit, and topology-label filtering are all exercised directly and none of these tests would pass if the corresponding method were a no-op. The `cmd/kube-proxy/app/server_test.go`/`server_linux_test.go` deletions (`Test_getNodeIPs`, `Test_waitForPodCIDR`, `TestProxyServer_platformSetup`) are adequately relocated into `TestNewNodeManager`, which actually covers more permutations (IP-missing vs. PodCIDR-missing precedence, node-not-found) than the originals.

However, there is one real, fairly serious hole, plus two moderate gaps that align with the task's own callouts (nil/old-object transitions, address changes, empty node).

## Critical Gaps

**1. `NodeConfig`'s informer→handler dispatch has zero test coverage, and silently diverges from its own doc comment — severity 8**
`pkg/proxy/config/config.go:260-299` (`NodeHandler` interface + `NewNodeConfig`). The doc comment states `OnNodeChange` "is called whenever **creation or modification** of node object is observed" (config.go:263-264), but `NewNodeConfig` only wires `UpdateFunc` and `DeleteFunc` — there is **no `AddFunc`** (config.go:288-294, compare to `NewServiceConfig`/`NewEndpointSliceConfig` a few lines above which do register `AddFunc`). This means a handler registered on `NodeConfig` (in production, only `s.NodeManager`, per `cmd/kube-proxy/app/server.go`) never receives `OnNodeChange` for a pre-existing node discovered via the informer's initial List — only for subsequent updates/resyncs. That happens to be masked in production because `NewNodeManager`'s constructor blocks and separately polls for the node before `NodeConfig.Run` even starts. But this dispatch mechanism — the actual "merge OnNodeAdd/OnNodeUpdate into OnNodeChange" glue that is the subject of this PR — has **never been exercised by any test**, before or after this PR (`grep -rn "NewNodeConfig(" --include="*_test.go"` returns nothing repo-wide). Contrast with `NodeTopologyConfig`, which is a nearly identical type and does have `TestNewNodeTopologyConfig` (config_test.go:486) exercising Add/Update through a fake watch. If `handleChangeNode`/`handleDeleteNode` (config.go:320-345) were changed to no-ops, or `AddFunc` were mistakenly wired later (reintroducing double-delivery), no test in the repo would catch it. This is the exact class of "would pass even if OnNodeChange were a no-op" risk the task asked about — it's just one layer up (the dispatcher, not the merged method itself, which is well tested in isolation).
- Suggested test: an analogue of `TestNewNodeTopologyConfig` for `NodeConfig` — register a mock `NodeHandler`, drive a fake watch through Add (of a pre-existing node), Update, and Delete, and assert `OnNodeChange`/`OnNodeDelete` are invoked (or not) as actually documented. This would either confirm the current Add-is-silently-skipped behavior is intentional (and prompt fixing the doc comment) or catch that it's a bug.
- Secondary, lower-priority item folded in here: the tombstone-recovery branch in `handleChangeNode` (config.go:322-330) and the "unexpected object type" `utilruntime.HandleError` paths in both `handleChangeNode`/`handleDeleteNode` are also untested — likely low-value since `DeletedFinalStateUnknown` shouldn't normally reach an `UpdateFunc` callback, so I'd rate that sub-item only 3.

## Important Improvements

**2. No negative test for `watchPodCIDRs=false` + PodCIDR change (the common-case gating branch) — severity 7**
`pkg/proxy/node.go:150-157` and `pkg/proxy/node_test.go:231-292` (`TestNodeManagerOnNodeChange`). All four table cases either combine "IPs differ, PodCIDRs untouched" (watchPodCIDRs implicitly false) or "watchPodCIDRs=true, PodCIDRs differ." None test **watchPodCIDRs=false with a real PodCIDR change**, i.e., the case that should produce `expectedExitCode: nil` despite `updatedPodCIDRs != initialPodCIDRs`. `watchPodCIDRs=false` is the default and applies to every cluster not using `LocalModeNodeCIDR` — the majority. If the `if n.watchPodCIDRs {` guard were ever accidentally dropped or inverted, every non-NodeCIDR-mode kube-proxy in a cluster would start crash-looping on PodCIDR churn, and this table wouldn't notice.
- Suggested test: add a case `{name: "PodCIDRs differ but watchPodCIDRs is false", initialPodCIDRs: [...], updatedPodCIDRs: [...different...], watchPodCIDRs: false, expectedExitCode: nil}`.

**3. `OnNodeChange` with an "address-losing" update (empty/invalid node) — severity 5**
`pkg/proxy/node.go:140-172`. On update, `n.node = node` is set unconditionally (line 145) *before* `GetNodeHostIPs(node)` is attempted (line 159). If the new node has no valid addresses, `GetNodeHostIPs` errors, the function logs and returns early (161-163) — but the manager's internal state has already been overwritten with the broken node, and `NodeIPs()` (node.go, `nodeIPs, _ := utilnode.GetNodeHostIPs(n.node)`) silently discards that error and returns an empty slice from then on. No test drives `OnNodeChange` with a node that has an empty/invalid `Status.Addresses` (the task explicitly calls out "empty node" as a case to check) to confirm this doesn't crash and to pin down the resulting `NodeIPs()`/`Node()` behavior.
- Suggested test: call `OnNodeChange(makeNode())` (no `tweakNodeIPs`, i.e., no addresses) after a manager has a valid initial state; assert no exit occurs, and assert `NodeIPs()` returns empty/nil rather than stale IPs — pin the actual (possibly surprising) contract.

**4. No test for topology-zone label removal — severity 5**
`pkg/proxy/config/config_test.go:486-597` (`TestNewNodeTopologyConfig`). Every scenario in this test only **adds/changes** the zone label upward (none → region-only → zone-set → zone-changed). There is no case where a node **loses** its `topology.kubernetes.io/zone` label (e.g., transitions from `{zone: "us-east-1b"}` back to `{}`), which should still fire `OnTopologyChange` with an empty map per the `reflect.DeepEqual` guard in `handleNodeEvent` (config.go, `pkg/proxy/config/config.go` ~line 500s). This is exactly the kind of edge transition the deleted `OnNodeUpdate`-based label tests implicitly covered by symmetry (labels going from populated → different, including empty).
- Suggested test: add a step removing `LabelTopologyZone` from the previous node and assert the handler is invoked with an empty `topologyLabels` map.

## Test Quality Issues

None rise to "brittle/overfit to implementation" — the `NodeManager`/`NodeTopologyConfig` tests assert on externally observable behavior (exit code invoked, resource version reflected in `Node()`, handler-received label map) rather than internal fields, and use realistic fake-informer/fake-watch plumbing rather than mocking away the code under test.

## Positive Observations

- `TestNewNodeManager` (`pkg/proxy/node_test.go:80-224`) is materially more thorough than the three tests it replaces (`Test_getNodeIPs`, `Test_waitForPodCIDR`, `TestProxyServer_platformSetup`), including the ordering nuance that a missing NodeIP is reported before a missing PodCIDR even when both are absent (`node.go` poll function checks IP before PodCIDR).
- `TestNodeManagerOnNodeChange` and `TestNodeManagerOnNodeDelete` correctly assert on the `exitFunc` side effect via dependency injection rather than the old `klog.OsExit`/`panic`-based `customExit` hack, which is a real improvement in test robustness (no longer relies on global mutable state / package-level `klog.OsExit`).
- `TestNewNodeTopologyConfig` (config_test.go:486) correctly distinguishes "irrelevant label added" (no notification) from "relevant label added" (notification), and correctly asserts that changing an *irrelevant* label while the relevant one is unchanged still does not re-notify (config_test.go:582-597) — a real behavioral assertion, not a tautology.
- `healthcheck_test.go`'s replacement of `hs.SyncNode(...)` with `nodeManager.OnNodeChange(...)` (healthcheck_test.go:897-972) exercises the real `NodeManager` rather than a bypassed field-setter, which is strictly more realistic than the old direct-mutation test and preserves the full taint/delete → 200/503 matrix from before.
- The `iptables`/`ipvs`/`nftables` proxier `OnNodeAdd`/`OnNodeUpdate`/`OnNodeDelete` → `OnTopologyChange` migration has no dedicated unit tests either before or after this PR (confirmed via repo-wide grep) — this is a pre-existing gap, not something this PR regressed, so it is not listed above as a PR-caused loss, but it is worth naming for awareness: `pkg/proxy/iptables/proxier_test.go`, `pkg/proxy/ipvs/proxier_test.go`, `pkg/proxy/nftables/proxier_test.go` never directly tested the node-label→`nodeLabels`/`topologyLabels` wiring at the proxier level (only `CategorizeEndpoints` is tested directly in `topology_test.go`).
